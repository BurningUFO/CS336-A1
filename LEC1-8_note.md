# Lecture 1: Overview and Tokenization

**基石一：基础架构 (Basics) —— Transformer 的现代魔改**

- **归一化**：用计算更轻量的 RMSNorm 替代 LayerNorm。
- **位置编码**：引入 RoPE（旋转位置编码）。
- **非线性激活**：把 MLP 里的 ReLU 换成了 SwiGLU。
- **注意力机制**：全量注意力极其消耗显存，所以引入了 MQA/GQA（分组查询注意力）来降低 KV Cache 的显存开销，甚至引入滑动窗口注意力（Sliding Window Attention）。

**基石二：系统与底层优化 (Systems) —— 跨越“内存墙”**

- 大模型真正的瓶颈根本不在于 ALU（算术逻辑单元）的计算速度，而在于极其拉胯的**数据搬运成本**。
	- **内存墙 (Memory Wall)**：GPU 的计算核心极快，但显存 HBM把数据运过来的带宽极慢。如果每次计算都去 HBM 读写，算力全浪费在等数据上了。
	- **内核融合 (Kernel Fusion)**：为了减少数据搬运，需要学习用 OpenAI 的 Triton 框架写底层 Kernel，把多个算子（比如乘法和激活函数）“融合”在一起，让数据在 GPU 的高速缓存（SRAM）里一次性算完再送回去。
- **推理瓶颈 (Inference)**：生成文字（Decode 阶段）是一次只能蹦一个词的自回归过程，极度受限于内存带宽。为了提速，业界发明了**投机解码 (Speculative Decoding)**，用一个小模型提前“猜”几个词，大模型并行验证。这在底层逻辑上，完美契合了现代 CPU 架构中的**分支预测 (Branch Prediction)** 机制！

**基石三：缩放定律 (Scaling Laws) —— 算力经济学**

- 在真实的工业界，算力（FLOPs 预算）是极其昂贵且固定的。面对一笔固定的算力，你是该训练一个“参数量大、但见识少”的模型，还是一个“参数量小、但饱读诗书”的模型？ DeepMind 的研究给出了极其暴力的工程经验法则——**Chinchilla 法则**： 最优的配比是，训练的 Token 数量应该是模型参数量的 **20 倍**。也就是说，如果你要训练一个 14 亿参数的小模型，最经济的算力利用方式是喂给它 280 亿个 Token 的数据。

**基石四：数据工程 (Data) —— 垃圾进，垃圾出 (GIGO)**

- 大厂在训练前沿模型时，实际上要花极大精力（甚至花重金购买高质量数据）来做预处理：
	1. **文本提取**：把 HTML/PDF 转化为纯文本，同时尽量保留结构信息。
	2. **过滤 (Filtering)**：训练分类器，把黄赌毒、无意义的 Spam（垃圾邮件）全洗掉。
	3. **去重 (Deduplication)**：极其关键！如果模型在训练集里看了 100 遍同一句广告词，它的权重就会严重过拟合。
- 3b1b 第 7 集里那个拥有 50,000 个神经元的 MLP（多层感知机）吗？它负责死记硬背世界知识。如果在这一步你不做去重和过滤，那些极其珍贵的神经元，就会在梯度下降中被白白浪费掉，拿去记忆互联网上的垃圾广告！

**基石五：🌟 核心难点：对齐微调 (Alignment) —— 注入灵魂**

- 用上述干净数据训练出来的叫 **Base Model（基础模型）**。如果你问基础模型“如何制造炸弹”，它会极其专业地给你生成维基百科式的步骤。它只是在做概率接龙，没有人类的道德观，也不懂“对话”的格式。 我们需要给它套上缰绳，这分为两步：
	1. **SFT (监督微调)**：人类手写几千到几万条高质量的“问答对”（Prompt-Response），让模型学会 Chatbot 的基本问答格式。
	2. **RL (强化学习)**： 为了进一步提升逻辑和安全性，引入基于反馈的学习。比如 **DPO (直接偏好优化)** 或近期 DeepSeek 爆火的 **GRPO (群组相对策略优化)**。让人类（或另一个验证器模型）给它的多个回答打分，强制模型在后续生成中，拉高“安全/好回答”的概率得分，压低“危险/废话”的得分。

**为什么不能用简单的分词方式？(The Tokenization Dilemma)**

- 为什么直觉上的分词方法都行不通：
	- **按字符/字节拆分 (Byte-based)**：比如把 "hello" 拆成 `h, e, l, l, o`。好处是词表极小（只有 256 个字节）。**致命缺点**：一个长句子会被拆成成千上万个字节！还记得 Transformer 的 Attention 机制是 $O(N^2)$ 的复杂度吗？序列长度一翻倍，显存占用直接变 4 倍，GPU 当场去世。
	- **按单词拆分 (Word-based)**：按空格拆分。**致命缺点**：遇到罕见词、拼写错误或者像中文这种没有空格的语言，词表大小会无限膨胀。遇到没见过的词只能输出 `<UNK>`（未知符号），导致信息严重丢失。

**BPE 字节对编码算法 (Byte-Pair Encoding)**

- 1994 年发明的无损数据压缩算法，成了今天大模型的救星。BPE 的核心思想是**“从底层字节开始，贪心地把最常见的组合粘在一块”**。
- **算法运行全景图**：
	1. **字节打底**：先把所有训练文本全部转化为最基础的单字节（UTF-8 编码，0-255），此时词表大小 `vocab_size = 256`。
	2. **暴力统计**：扫描全量文本，统计所有**相邻字节对**的出现频率。
	3. **贪心合并**：找到出现频率最高的那一对，将它们和成一个新的 Token ID（比如 `256`），并将词表扩大 + 1。然后在文本里把所有的 `t` 和 `h` 替换成这个新 ID。
	4. **疯狂套娃**：重复上述步骤。接下来最高频的可能是刚才合并出来的 `th` 和 `e`，于是把它们再合并成 `the`（分配 ID `257`）。
	5. **结束条件**：一直循环合并，直到达到你预设的词表大小（例如 GPT-2 的 50,257）。

**工程深水区 —— 时间复杂度陷阱**

- 虽然 BPE 原理听起来很简单，但在真正写代码时，如果每合并一次，你都去重新 `for` 循环遍历几十 MB 的大数组来统计词频，你的时间复杂度将极其恐怖。 真正的工业级 BPE（比如 OpenAI 的 `tiktoken` 库），底层必定动用了更高级的数据结构。你需要在循环中**动态维护计数器**（利用哈希表 dict），甚至使用优先队列（Priority Queue），避免重复扫描未发生变动的子串。

#  Lec. 2: Pytorch, Resource Accounting 	

算力评估公式： 总计算量 (FLOPs) $\approx 6 \times \text{Parameters} \times \text{Tokens}$ 算出总 FLOPs 后，除以显卡的实际吞吐量（考虑到 MFU 算力利用率通常只有 50% 左右）

**浮点数精度 (Data Types)**

- 在 C++ 里我们习惯无脑用 `float`（32位，4字节）。但在动辄几十 GB 显存的大模型里，这就太奢侈了。
	- **FP32 (Float32)**：标准单精度。安全，但吃内存。
	- **FP16 (Float16)**：直接砍半（2字节）。**致命缺陷**：动态范围太小，遇到深度学习里极小的梯度（比如 $10^{-8}$）会直接“下溢”变成 0，导致模型崩溃（NaN）。
	- **🌟 BF16 (Brain Float 16)**：它砍掉了小数点的精度，但保留了和 FP32 一样宽广的指数范围（动态范围）。它既省了一半内存，又不会导致梯度爆炸或下溢。

**张量的底层内存与视图 (Tensors, Strides, & Contiguity)**

- 在 PyTorch 里，一个多维矩阵（Tensor）其实是个骗局。它的底层根本不是多维数组，而是一段**极其扁平的、一维的 C++ 连续内存数组 (Storage)**。

- PyTorch 是怎么把一维数组变成矩阵的？靠的是元数据 `Stride`。比如一个 $4 \times 4$ 的矩阵，横着走一步（列+1），底层内存指针跳 1 格（Stride=1）；竖着走一步（行+1），底层指针跳 4 格（Stride=4）。

- **视图 (View) 与 连续性 (Contiguous)**：

	当你对 Tensor 执行 `transpose()`（转置）时，PyTorch **绝对不会去复制内存**，它只是把横向和纵向的 Stride 值对调了一下！这叫作生成了一个**视图 (View)**。

	但这样做会导致张量在内存中变成**非连续的 (Non-contiguous)**。如果你紧接着对这个非连续张量执行某些需要内存对齐的算子（比如 `.view()`），程序会直接原地爆炸报错！此时必须调用 `.contiguous()`，强迫底层的 C++ 引擎老老实实地开辟一块新显存，把数据按顺序深拷贝（Deep Copy）过去。

**Einops**

- 在传统的 PyTorch 代码中，这种极其丑陋且反人类的写法：`x.transpose(-1, -2)` 或者 `x.view(b, -1, h)`。只要你的张量多加了一个维度，这些硬编码的 `-1` 和 `-2` 就会立刻全线崩溃，导致极其隐蔽的 Shape Mismatch（形状不匹配）Bug。
- **Einops (爱因斯坦操作) 的降维打击**： 它受爱因斯坦求和约定的启发，允许你直接用**字符串变量名**来操纵内存视图！
	1. **`rearrange` (重排/拆解)**：如果你要把注意力机制中的多头（Heads）拆开，不用算数算半天，直接写： `rearrange(x, 'batch seq (heads head_dim) -> batch seq heads head_dim', heads=8)` 底层的内存步长（Stride）它自动帮你搞定。
	2. **`einsum` (超级矩阵乘法)**：计算 Q 和 K 的点积时，不用再调用容易搞混维度的 `torch.matmul`： `einsum(q, k, 'batch seq1 hidden, batch seq2 hidden -> batch seq1 seq2')` 凡是没有出现在箭头右边的维度（`hidden`），全部会被自动执行内积求和。
- 使用原始 PyTorch 就像在使用 `void*` 指针加上硬编码的偏移量（Offset）去读写内存，极度危险；而引入 `Einops`，就像是给这块内存套上了一个强类型的具名 `Struct`，所见即所得，代码即注释。

**什么是 FLOPs？**

- **FLOPs (小写 s)**：Floating Point Operations，指**浮点运算总次数**，代表工作总量（相当于代码的时间复杂度 $O(N)$）。
- **FLOP/s (也常写为 FLOPS，大写 S)**：Floating Point Operations Per Second，指**每秒浮点运算次数**，代表硬件的计算速度（即显卡的马力）。

- 在大厂做基建，必须对 H100 等显卡的标称算力（比如 1979 Teraflop/s）有肌肉记忆，这是后续评估模型训练时长的基石。

**矩阵乘法 (Matmul) 到底消耗多少算力？**

- 大模型里 99% 的计算量都来自矩阵乘法。如何手算一个矩阵乘法的 FLOPs？

	假设输入矩阵 $X$ 形状为 $(B, D)$，权重矩阵 $W$ 形状为 $(D, K)$。

	为了算出输出矩阵 $(B, K)$ 中的**某一个具体元素**，你需要把 $X$ 的一行（长度 $D$）和 $W$ 的一列（长度 $D$）进行点积。

	- 点积的底层动作：$D$ 次乘法 $+$ $D$ 次加法 $\approx 2D$ 次浮点运算。
	- 输出矩阵一共有 $B \times K$ 个元素。
	- 总 FLOPs $= 2 \times B \times D \times K$。

- **“矩阵乘法的计算量，永远等于 2 乘以参与运算的三个核心维度的乘积。”**

**算力利用率 (MFU)**

- NVIDIA 官方宣传 H100 的算力是 1979 TeraFLOP/s，但这只是理论上的峰值（在无视数据搬运、完美并行的真空环境下）。

	Percy 教授引入了一个极其关键的工程指标：**MFU (Model FLOPs Utilization，模型算力利用率)**。

	$$MFU = \frac{\text{实际观测到的计算速度}}{\text{硬件标称的峰值速度}}$$

- **工程现实**：一个写得极其优秀的底层 CUDA/C++ 代码，MFU 也很难超过 0.5（50%）。如果你写的代码 MFU 只有 5%，说明 GPU 绝大部分时间都在干等着数据从显存搬运到寄存器，这是严重的性能事故！所以在跑大规模任务前，必须先 Profiling（性能基准测试）算一下你的 MFU。

**反向传播的算力法则 ($6NDP$)**

- 前向传播（Forward Pass）只是大模型计算的冰山一角。 为什么跑训练比跑推理慢得多？我们来算一笔微积分的账：

	假设有一个矩阵乘法 $Y = XW$。其中 $X$ 是输入（激活值），$W$ 是权重。

	1. **前向传播 (Forward)**：只算一次 $X \times W$。根据我们刚才推导的公式，计算量是 $2 \times \text{Tokens} \times \text{Parameters}$。
	2. **反向传播 (Backward)**：根据链式求导法则，这里需要算**两次**大型矩阵乘法！
		- 第一步：求权重的梯度 $dW = X^T \times dY$（为了更新当前层的参数）。计算量同样是 $2 \times \text{Tokens} \times \text{Parameters}$。
		- 第二步：求输入的梯度 $dX = dY \times W^T$（为了把误差继续往上一层传！如果你不传给上一层，链式法则就断了）。计算量又是 $2 \times \text{Tokens} \times \text{Parameters}$。

- 前向传播需要 $2$ 份算力，反向传播需要 $4$ 份算力。**训练一个大模型的总算力消耗 $= 6 \times \text{Tokens} \times \text{Parameters}$**。

**参数初始化 (Initialization) 的防爆机制**

- 在 C++ 里，如果你声明了一个 `float` 数组但不赋初值，它里面会是乱码。在 PyTorch 里，如果你用标准正态分布随机初始化成千上万维的权重矩阵，由于多维矩阵疯狂相乘，方差会呈指数级爆炸，最终导致数值溢出（`NaN`）。
- **Xavier 初始化**：必须在初始化时，把随机生成的数值除以 $\sqrt{\text{输入维度}}$。这个简单的除法，是保障深度神经网络在刚开始训练时不至于原地爆炸的物理学基石。

**显存黑洞 (Memory Accounting)**

- 当你以为一个 70B（700亿参数）的模型，按 BF16（2字节）算，只需要 140GB 显存就能训练了，其实是错误的

	在训练时，除了模型本身的权重（Parameters），你的显存里还必须塞下另外三块吃内存的部分：

	1. **梯度 (Gradients)**：每个参数都需要一个对应的梯度变量来保存方向，大小等于参数量。
	2. **优化器状态 (Optimizer States)**：如果你用大名鼎鼎的 Adam 优化器，它需要为每个参数保存一阶动量（平均值）和二阶动量（方差）。而且为了保证精度，优化器状态通常必须用 **FP32（4字节）** 存储！这意味着每个参数的优化器状态就要吃掉 8 个字节！
	3. **激活值 (Activations)**：前向传播算出来的中间结果 $X$，**绝对不能丢！** 因为你在反向传播算 $dW = X^T \times dY$ 的时候，还需要用到它！层数越深、序列越长（Batch Size $\times$ Sequence Length），保留的激活值就把显存撑得越大。

- 所以训练时所需的显存，往往是模型本身大小的 **4 到 6 倍**甚至更多。

# Lec. 3: Architectures, Hyperparameters

**架构变迁一：从 Post-Norm 到 Pre-Norm (前置归一化)**

- 在 2017 年原版的 Transformer 中，LayerNorm (层归一化) 是放在残差连接（Residual Stream）之后的，这叫 Post-Norm。 **致命缺陷**：把它放在残差流里，就像在 C++ 的主线程通信管道上加了一个强行拦截的过滤器，极大地阻碍了梯度的反向传播，导致训练深层网络时极不稳定（经常出现 Loss 爆炸和梯度尖峰）。
- 现在所有的前沿模型（除了极少数由于 Bug 写错的），都把归一化挪到了进入 Attention 或 MLP **之前**。 这样残差流（主管道）保持绝对畅通，从第一层直通最后一层，成为了训练千亿参数模型的顶级“稳定器”。

**架构变迁二：Layer Norm 进化为 RMS Norm**

- 传统的 Layer Norm 需要算两步：先减去均值（Mean），再除以标准差（Standard Deviation），还要加上一个可学习的偏置 $\beta$。

	现代模型（如 LLaMA）全面换成了 **RMS Norm (均方根归一化)**。它的改动极其残暴：**直接不减均值了，也不加偏置 $\beta$ 了！** 纯粹只除以一个均方根。

- **C++ 底层/系统性能透视**：

	如果你只看浮点运算次数 (FLOPs)，这个改动微乎其微（只省了 0.17% 的计算量）。但教授说的 **“内存移动 (Memory Movement)”** 却极其消耗计算：在 GPU 底层，不减均值、不加偏置，意味着系统**少了一次全局求和同步**，且**少从显存里读取了一大块参数变量**。在 GPU 极其拉胯的显存带宽面前，省下搬运，直接让运行速度获得了实打实的提升，而且模型最终的 Loss 表现完全没有变差！

**架构变迁三：彻底抛弃 Bias (偏置项)**

- 以前学线性层（Linear Layer），我们倒背如流的公式是 $y = xW^T + b$。但如果你去看现代开源大模型（如 LLaMA）的源码，你会发现：**所有的偏置项 $b$ 全被删了！**
- **为什么这么干？** 
	- **稳定性**：经验表明，丢掉 Bias 能让超大规模矩阵相乘时的数值更稳定。
	- **效率**：纯粹的矩阵乘法 $y = xW^T$ 对硬件和底层 Kernel（如 cuBLAS）来说是最容易做到极致优化的，加个尾巴反而碍事。

**核心进化一：从 ReLU 到 门控线性单元 (GLU)**

- 在经典的神经网络中，MLP (多层感知机) 的逻辑非常简单直接：`输出 = W2(ReLU(W1(x)))`。 但 Google 的研究人员发现，如果引入**门控机制 (Gating)**，效果会出奇的好。什么是 GLU（Gated Linear Unit）？ 它不再是简单地通过一个激活函数，而是增加了一条**平行的分支**（需要多学习一个权重矩阵 $V$）。 逻辑变成了：先把输入 $x$ 分别通过矩阵 $W_1$ 和 $V$。对 $W_1$ 的结果做激活，然后与 $V$ 的结果做**按位乘法 (Element-wise multiplication)**。
- **SwiGLU 的诞生**：如果在这个门控机制里，使用的激活函数是 **Swish**（公式为 $x \cdot \sigma(x)$），那这个整体架构就叫 **SwiGLU**。目前 LLaMA、PaLM 等几乎所有顶流模型，全都在用它！

**核心进化二：算力账本的平衡术 (Scaling down by 2/3)**

- GLU 引入了一个全新的权重矩阵 $V$！这意味着参数量和内存读取量直接暴增，这不符合我们追求极致 Efficiency（效率）的原则。
- **解决方案**：
	- 为了保证使用 SwiGLU 的模型和原来使用 ReLU 的模型**参数量绝对相等**（做到公平比较），架构师们做了一个极其暴力的数学操作——**把 MLP 中间隐藏层的维度直接乘以 $\frac{2}{3}$**。
	- 在代码层面，原本隐藏层维度是输入维度的 $4$ 倍；换成 SwiGLU 后，隐藏层维度通常被强制压低到了 $2.66$ 倍左右（比如 $4 \times \frac{2}{3}$）。这样一来，参数总账刚好平了，但模型的表达能力却神奇地变强了！

**核心进化三：串行 vs. 并行网络层 (Serial vs. Parallel Layers)**

- 在标准的 Transformer 里，数据流是**串行 (Serial)** 的：先过 Attention 层，把结果加到残差流里；然后再过 MLP 层，再把结果加到残差流里。
- **并行架构 (Parallel)**：像 GPT-J 和 PaLM 这样的模型，为了系统效率，把拓扑结构改了。 它们让数据**同时**进入 Attention 和 MLP！两个模块各算各的，算完之后，把两者的结果一次性全部加到残差流里。
-  这就极其像 C++ 里的多线程并发编程。串行模式下，MLP 线程必须阻塞等待 Attention 线程执行完毕（有数据依赖）；而在并行模式下，两个线程同时发射，互不干扰，最后用一个 `join()` 合并结果。这样不仅算力利用率更高，还能在底层把两个矩阵乘法的 Kernel（内核）合并（Fusion），极大地节省了显存带宽！

## 旋转位置编码 RoPE

**架构演进：从绝对位置到相对位置**

- 由于 Transformer 是同时吞下整个句子的，它天生是“语序色盲”。 在早期（比如 GPT-2 或原版 Transformer），做法是在最底层的词向量上直接加上一个代表位置的“绝对位置向量”（Absolute Embeddings）。但这不符合语言直觉：我们在读句子时，关注的往往是两个词之间的**相对距离**，而不是它们具体在第几个字。

**RoPE 的几何与数学直觉**

- 我们希望词向量 $x$（在位置 $i$）和词向量 $y$（在位置 $j$）的点积，仅仅取决于它们的相对距离 $i-j$。

	数学家们想到了一个绝妙的几何特性：**旋转**。

	如果你在二维坐标系里有两个向量，不管你把它们统一顺时针旋转多少度，它们之间的**相对夹角（点积）是绝对不会变的**！

	因此，我们只需让处于位置 $i$ 的词向量旋转 $i \times \theta$ 度。这样，当 $Q$（查询）和 $K$（键）进行内积相乘时，数学公式会自动抵消掉绝对位置，完美保留下它们之间相差的相对度数 $(i-j) \times \theta$！

**如何旋转高维向量？**

- 由于我们的词向量是 12288 维的超高维空间，怎么做二维旋转？

	极其暴力的工程拆解：**把 12288 维切成 6144 个二维对 (Pairs)**！

	对于每一对二维向量，乘以一个经典的二维旋转矩阵（包含 $\sin$ 和 $\cos$）。不同的维度配对赋予不同的旋转速度（高频负责近距离，低频负责长距离）。

- **C++ / 底层开发思维映射**：

	**注意！RoPE 的生效位置与老模型完全不同！**

	在老模型中，位置编码是在最底层刚输入时加进去的；而 RoPE 是在 **Attention 模块内部**生效的。

	在执行 $Q \times K^T$ 之前，你必须在代码里拦截住 $Q$ 和 $K$ 的张量，分别对它们执行旋转变换：

	`Q_rotated = apply_rope(Q)`

	`K_rotated = apply_rope(K)`

	然后再让它们相乘。在这部分写代码时，为了追求极致效率，千万不要傻傻地去构造庞大的稀疏旋转矩阵来做矩阵乘法。在 PyTorch 中，这会被优化为奇偶数切片操作（比如 `x[..., 0::2]` 和 `x[..., 1::2]`）和元素的直接加乘（Element-wise operations）。如果这步张量切片写错一个步长 (Stride)，整个模型的语序就会彻底错乱。

## 超参数（Hyperparameters）

**黄金法则一：MLP (前馈网络) 到底该多宽？**

- 在 Transformer 中，词向量（Dimension, 记为 $D$）进入 MLP 时，会被投影到一个更宽的维度（Feed-Forward Dimension, 记为 $D_{FF}$），然后再投射回来。

	这个 $D_{FF}$ 该定多大？

	- **ReLU 时代的共识**：**$4 \times D$**。比如输入是 1024 维，中间层必定是 4096 维。这几乎是原版 Transformer 传下来的铁律。
	- **GLU 时代的共识**：上一节我们讲过，为了在使用 SwiGLU 时保持参数量不变，这个比例被压缩到了 **$\frac{8}{3} \times D \approx 2.66 \times D$**。

- **例外（T5 模型）**：Google 的 T5 模型极其暴躁，它把中间层直接拉宽了 **64 倍**！虽然效果也不错，但后来他们自己出的升级版（T5v1.1）又老老实实地退回了 $2.66$ 的标准比例。这说明，瞎调宽度的收益极低，遵守共识才是最高效的。

**黄金法则二：多头注意力（Heads）怎么分？**

- 如果你的词向量维度 $D = 1024$，你要分多少个注意力头（Heads）？每个头处理多少维度（Head Dimension）？
	- **绝对共识**：无论模型怎么变，**每个 Head 的维度通常被死死锁在 64 或 128**。
	- **推导**：这意味着，$D = \text{Num\_Heads} \times \text{Head\_Dim}$。如果规定每个头处理 64 维，那 1024 维的模型就必定分配 $1024 / 64 = 16$ 个头。
- 为什么非要是 64 或 128？这是因为 GPU 底层 Tensor Core（张量核心）的硬件架构，对这两个数字的矩阵乘法优化得最好。如果你硬要设成 73 维，底层对不齐（Memory Misalignment），显存读写效率会直接血崩！

**黄金法则三：模型的“纵横比” (Aspect Ratio)**

- 给定一定的参数预算，你是该建一个“又深又窄”（层数极多，维度很小）的模型，还是一个“又浅又宽”（层数很少，维度极大）的模型？
	- **指标**：$\text{纵横比} = \frac{\text{模型维度 (Width)}}{\text{层数 (Depth)}}$
	- **Kaplan 实验结论**：通过大量的烧钱实验，大家发现最佳的纵横比大约在 **100 到 150 之间**。 比如，维度 $D=12288$，层数 $N=96$（GPT-3 的配置），纵横比就是 $128$。在这个区间内，模型性价比最高。
- 在做分布式训练时，太深的模型很难做流水线并行（Pipeline Parallelism，延迟太高）；太宽的模型很难做张量并行（Tensor Parallelism，通信带宽吃不消）。128 左右是一个极其完美的系统妥协点。

**黄金法则四：词表大小 (Vocabulary Size)**

- 还记得上节课咱们手推过的 BPE 分词算法吗？词表（Vocab Size）该设多大？
	- **早期（单语种）**：30,000 到 50,000 足矣（如 GPT-2 的 50,257）。
	- **现代（多语种/代码）**：像 LLaMA 3 或 Qwen 这样的工业级模型，词表大小全面飙升到了 **100,000 到 250,000** 之间。
- **为什么变大？** 词表越大，代表一个汉字或一段 Python 代码能被压缩成更少的 Token。 在推理生成阶段，模型每次只能蹦一个 Token；如果你用大词表，模型说同一句话需要的步数就更少，用户的等待时间就大幅缩短，算力成本也随之骤降！

## 训练技巧与 Softmax 

**核心反转一：为什么只看一遍数据的模型，还需要 Weight Decay？**

- 这是深度学习里最反直觉的一幕。在传统的机器学习中，我们用 Dropout 和 Weight Decay（权重衰减/L2正则化）是为了防止模型在同一批数据上“过拟合 (Overfitting)”。 **悖论**：大模型训练的语料库浩如烟海，模型终其一生只能把数据看一遍（1 Epoch），根本不存在过拟合的可能！那为什么大家都在配置文件里加上了 Weight Decay？
- 教授揭秘，现在加 Weight Decay **根本不是为了正则化，而是为了操纵优化器的动力学 (Optimization Dynamics)！** 实验发现，Weight Decay 配合余弦学习率衰减（Cosine Learning Rate Decay），会在训练的最后阶段（学习率降到接近 0 时），让训练 Loss 发生一次极其剧烈的“高台跳水”（陡降）。如果不加 Weight Decay，这个极限的优化红利就吃不到。

**核心反转二：梯度爆炸的元凶与 Z-loss**

- 你在跑几十亿参数的模型时，如果突然满屏爆红，提示梯度为 `NaN`（Not a Number），元凶 99% 是 **Softmax**。

	Softmax 底层要算 $e^x$，还要除以一个分母（归一化常数 $Z$）。在 C++ 或 CUDA 底层，浮点数指数运算极易溢出（Overflow），除法极易碰到除以 0 的情况。

- **Z-loss 镇压法**：

	PaLM 等前沿模型发明了一个极其聪明的辅助损失函数——**Z-loss**。

	它在最终的 Loss 公式里硬生生加了一项：$\alpha \cdot \log^2(Z)$。

	这个公式的物理意义就是：**强迫 Softmax 的分母 $Z$ 必须呆在 1 附近！** 只要分母安全，底层的除法和对数运算就绝对安全，梯度尖峰（Gradient Spikes）瞬间消失。

**核心反转三：QK Norm (给注意力戴上紧箍咒)**

- 除了输出层，模型里还有一个地方全是 Softmax，那就是 Attention 矩阵计算 $Q \times K^T$ 之后！

	当特征维度 $D$ 很大时，$Q$ 和 $K$ 的点积数值会变得极其庞大，直接把 Softmax 撑爆。

- **物理隔离**：为了防止这种惨剧，现代架构（如 Gemma 2）引入了 **QK Norm**。

	在计算 $Q \times K^T$ 之前，先强行用 Layer Norm 把 $Q$ 和 $K$ 洗一遍，把它们的数值规模死死卡在安全范围内。这就像在两个高压电极之间加了一个稳压器，彻底杜绝了注意力矩阵的数值短路。

# KV Cache 与 MQA

**算术强度 (Arithmetic Intensity)**

- 这是计算机体系结构里极其核心的概念。

	$\text{算术强度} = \frac{\text{计算量 (FLOPs)}}{\text{内存访存量 (Bytes)}}$

	- **训练时**：我们在做庞大的矩阵乘法，计算量极大，访存量相对较小（算术强度高），这叫**算力瓶颈 (Compute-bound)**，GPU 跑得很开心。
	- **推理时**：模型每次只能往外蹦一个词 (Token)。为了算这一个词，系统必须把整个模型几百 GB 的权重从显存搬到计算核心里！计算量极小，但访存量极大（算术强度极低），这叫**内存瓶颈 (Memory-bound)**。GPU 大部分时间都在无聊地等数据传输。

**KV Cache (键值缓存)**

- 在生成第 100 个词时，前面 99 个词的特征其实不需要重新算一遍。

	我们在代码里开辟一块显存空间，把历史 Token 算出来的 Key (键) 和 Value (值) 永远缓存下来。当生成新词时，只需要算出当前这个词的 Query ($Q$)，去和缓存里的历史 $K$ 做点积，然后再去乘缓存里的历史 $V$ 即可。

- 在写底层代码时，KV Cache 的本质是一个**动态增长的多维数组**。随着对话越来越长，这个数组的长度 $N$ 会不断递增。如果你每次增加一个词都去 `realloc` (重新分配内存) 并做深拷贝，系统会直接卡死。所以，你必须像 C++ 的 `std::vector` 底层一样，提前预分配一块巨大的连续内存 (Max Context Length)，然后用一个指针记录当前写到了哪里。

**从 MHA 到 MQA / GQA**

- KV Cache 解决了重复计算的问题，但引入了一个更致命的怪物：**显存带宽杀手**。每个注意力头 (Head) 都要存一份自己的 $K$ 和 $V$。如果上下文特别长，缓存大小会瞬间吃光你的 GPU 显存。
	- **MHA (多头注意力)**：传统的做法，比如 32 个 Query 头，对应 32 个 Key 头和 32 个 Value 头。
	- **MQA (多查询注意力)**：极其暴力的优化！ 32 个 Query 头，**强行共享 1 个 Key 头和 1 个 Value 头**！这样 KV Cache 的体积直接缩小了 32 倍！GPU 访存压力瞬间释放，推理速度狂飙。
	- **GQA (分组查询注意力)**：MQA 太极端了，可能会导致模型变笨。于是 LLaMA 2 等模型采取了折中方案 GQA：把 32 个 Query 头分成 8 组，每 4 个 Query 共享 1 个 Key 和 1 个 Value。完美平衡了准确率和推理速度。

**稀疏与交替注意力 (Llama 4 的无限上下文黑科技)**

- 为什么现在的模型动辄支持百万字的长文本（Long Context）？ 除了 RoPE 的外推，架构师们还用了**滑动窗口 (Sliding Window)**。 在大多数层里，Token 只和它前面附近的一小块 Token 做注意力计算（节省海量显存）；只有在特定的少数几层（比如每隔 4 层），才做一次纵览全局的全量注意力。通过这种极其精妙的“局域+全局”积木拼接，模型在保持高智商的同时，极大地降低了系统负荷。 

#  Lecture 4: Mixture of experts

## MoE 的动机与基础路由机制

**算力墙与稠密模型 (Dense Models) 的死局**

- 前几节课我们推导过一个黄金公式：前向传播的 FLOPs $\approx 2 \times \text{Tokens} \times \text{Parameters}$。

	这意味着，对于传统的**稠密模型 (Dense Model，比如 LLaMA)**，输入的每一个词，都必须和模型里的**每一个**权重矩阵进行一次相乘。如果你想造一个万亿参数的巨兽来提升智商，计算量将庞大到没有任何芯片集群能跑得动，推理成本也会直接破产。

- 能不能让模型拥有万亿参数的“知识储备”，但在处理某个具体的词（比如“苹果”）时，只调用那部分懂“水果”或“科技公司”的参数，而让其他参数“休眠”？

**条件计算 (Conditional Computation)**

- MoE 的核心思想就是“条件计算”。

	我们不对 Attention 层开刀（Attention 依然是所有词全局共享的），我们只**魔改 MLP（前馈网络）层**！

	我们把原本极其庞大的单个 MLP 模块，拆分成 $E$ 个小型的 MLP 模块。这 $E$ 个小模块，就被称为 **“专家 (Experts)”**。比如设定 8 个专家，每个专家其实就是一个标准的 SwiGLU 结构。

**路由器 (Router / Gating Network)**

- 现在有 8 个专家，当一个词的隐藏层向量 $x$ 走过来时，该交给谁处理？

	我们需要引入一个全新的组件：**路由器 (Router)**。

	1. **打分**：Router 本质上就是一个极小的线性层 $W_r$。它把高维的输入 $x$ 投射成一个长度为 8 的向量（Logits），代表 8 个专家的得分。
	2. **概率化**：经过 Softmax 函数，把得分变成概率分布（比如专家 1 概率 0.8，专家 3 概率 0.15……）。
	3. **Top-K 选择**：大模型通常采取 **Top-2 路由**。也就是说，强制只把 $x$ 发给概率最高的 2 个专家去进行繁重的矩阵运算。其他 6 个专家直接忽略（相乘结果记为 0）。
	4. **加权加和**：把选中的 2 个专家的输出结果，按照刚才算出的概率（比如 0.8 和 0.15）加权求和，送入下一层。

**动态分支**

- 从 C/C++ 编程的角度看，传统的 Dense 模型是线性的顺序执行，极度适合 GPU 这种擅长大规模矩阵运算的 SIMD（单指令多数据流）架构。 但 MoE 的 Router 引入了什么？**动态的分支判断 (`if/else` 或 `switch`)！** GPU 最讨厌 `if/else` 分支分歧（Branch Divergence）。Token A 去了专家 1，Token B 去了专家 4，这会导致内存读取变得极其零碎（Sparse Memory Access）。如何在底层把这些去向不同专家的 Token 重新打包成连续的内存块发送过去，是实现 MoE 代码时最折磨人的地方。

## MOE : 负载不均和辅助损失

**路由器的“马太效应” (Load Imbalance)**

- 假设你有 8 个专家。在模型刚初始化时，Router 的权重是随机的。恰好，专家 1 的初始状态对某些 Token 的预测稍微好了一丁点。 Router 发现把数据发给专家 1 能让 Loss 变低，于是它开始**疯狂地把越来越多的 Token 全塞给专家 1**。 **恶性循环**：专家 1 因为吃到了最多的数据，被训练得越来越好；而专家 2 到 8 因为接不到数据，梯度根本无法更新，变成了“死神经元”。最终，万亿参数的 MoE 退化成了一个只有 1/8 参数生效的残废模型！

**专家容量与 Token 丢弃 (Expert Capacity & Dropping)**

- 在 GPU 的 CUDA 内核里，最忌讳的就是动态分配内存。你不能对专家 1 说：“嘿，这批有 1000 个 Token 找你，我用 `malloc` 临时给你开个大数组”。GPU 要求内存在编译或初始化时就必须是**静态、连续、对齐**的固定大小！

- **物理封锁线 (Capacity Factor)**：

	为了迁就 GPU，架构师设定了**专家容量 (Expert Capacity)**。

	假设一个 Batch 有 800 个 Token，8 个专家。理论上每个专家分 100 个。我们设定一个容量因子（比如 $1.25$），那么底层在显存里为每个专家写死的物理数组大小就是 $125$。

- **Token Dropping (丢弃机制)**：

	如果 Router 发疯，一次性给专家 1 发了 150 个 Token 怎么办？前 125 个正常进入矩阵乘法计算；**多出来的 25 个 Token，直接被无情抛弃 (Dropped)！** 它们不经过任何 MLP 处理，直接通过残差连接（Residual Connection）原封不动地流向下一层。这对模型的智商是极大的打击。

**辅助损失函数 (Auxiliary Loss)**

- 既然强行丢弃 Token 会让模型变笨，那我们能不能在源头上“逼迫” Router 乖乖做到雨露均沾？

	答案是：在最终的交叉熵 Loss 之外，额外加一个惩罚项——**辅助损失 ($L_{aux}$)**。

- **数学直觉**：这个 $L_{aux}$ 的公式会同时计算两个东西：

	1. 每个专家实际接收到的 Token 比例（比如专家 1 收了 90%）。

	2. Router 给每个专家打出的平均概率得分。

		如果这两项的分布极度不均匀（方差极大），$L_{aux}$ 就会输出一个极其巨大的惩罚值，导致总 Loss 飙升。通过反向传播，Router 的权重会被强行扭转：**“不许全发给专家 1！就算发给专家 3 效率稍低，你也必须给我分摊出去！”**

## 细粒度切割与现代 MoE 的进化

**从“巨无霸”到“细粒度专家” (Fine-grained Experts)**

- 以前的 MoE，比如总参数量一定，切分成 8 个“大专家”，每次激活 2 个（Top-2）。

	研究人员灵机一动：如果总参数量保持绝对不变，我把它切分成 **64 个“小专家”**，每次激活 **8 个** 呢？

	总激活的计算量（FLOPs）一模一样！但是，组合爆炸了！

	原本的路由选择空间是 $\binom{8}{2} = 28$ 种组合；现在的选择空间变成了惊人的 $\binom{64}{8} \approx 4.4 \times 10^9$ 种组合！这种“细粒度化”让模型能以极其灵活的积木拼装方式，针对极其细分的知识点（比如“Python 并发编程”或“微积分公式”）调动最专属的参数，智商瞬间飙升。

- 从软件工程角度看，切碎是好事；但从 GPU 底层看，这是巨大的灾难。

	以前启动 2 个大矩阵乘法（GEMM），GPU 跑得很满；现在你要循环启动 8 个极小的矩阵乘法，这会引发严重的 Kernel 启动开销（Kernel Launch Overhead），导致 GPU 算力利用率（MFU）暴跌。为了解决这个问题，底层的 CUDA 工程师必须手写 **Grouped GEMM（分组矩阵乘法）** 算子，把这 8 个碎任务打包成一条指令塞进 GPU。

**剥离“共享专家” (Shared Experts)**

- 这是 DeepSeek-MoE 等架构提出的神级洞察。 在处理语言时，有大量的基础知识是**所有 Token 都需要的**（比如标点符号、基础语法、或者像 "the", "is", "a" 这种高频词）。如果让 Router 每次都去给这些基础词汇分配专家，不仅浪费路由器的算力，还会严重挤占某些特定专家的容量（导致严重的负载不均）。
- **物理隔离方案**：干脆在架构里划出一块特殊的自留地——**共享专家 (Shared Experts)**。 不管 Router 怎么分配，这几个共享专家**永远保持激活**，对所有 Token 生效，专门负责处理“标点语法”和“公共常识”；而 Router 只负责把剩下的配额（比如剩下的 6 个激活位），精确地分配给处理特定专业知识的路由专家（Routed Experts）。

**抛弃辅助损失 (Auxiliary-Free Load Balancing)**

- 为了防止专家被撑爆，我们加了辅助损失函数 $L_{aux}$。

	但现在的研究发现，$L_{aux}$ 是有毒的！它为了强行逼迫 Router 做到负载均衡，甚至会迫使 Router 把“C++ 指针”相关的 Token 发给“擅长做西红柿炒鸡蛋”的专家。虽然系统不 OOM（爆显存）了，但模型的准确率被严重破坏了。

- **现代解法（动态偏置 Dynamic Bias）**：

	直接砍掉 $L_{aux}$！我们在 Router 打分（Logits）的时候，偷偷给每个专家加一个可以动态调整的偏置项（Bias）。如果发现某个专家快满载了，就不在 Loss 上做文章，而是直接在底层把它的 Bias 调低一点点，让后来的 Token 稍微觉得它“没那么香”了。

## MoE 的物理边界：显存、并行与推理瓶颈

**“显存黑洞”**

- 虽然 MoE 每次只激活 2 个专家（FLOPs 很低），但你必须把 **所有 8 个（甚至 64 个）专家** 的权重全部塞进显存里！ 因为 Router 可能会把下一个词发给任何一个专家。如果你现从内存或 SSD 往显存里搬权重，那速度会慢到让人绝望（这叫 IO 瓶颈）。
- 一个 8x7B 的 MoE 模型，名义参数量接近 50B。按 BF16 算，光存模型就要 100GB 显存。目前主流的 H100 只有 80GB，单卡连模型都装不下，更别提跑训练和推理了。

**专家并行 (Expert Parallelism)**

- **C++ / 底层分布式思维映射**：既然单卡装不下，我们就得搞“分布式”。
	1. **切分**：把 8 个专家拆开，专家 1-4 放在服务器 A，专家 5-8 放在服务器 B。
	2. **全网路由 (All-to-All Communication)**：这是最硬核的一步！ 当服务器 A 处理的 Token 发现自己该去专家 5 时，系统必须通过极其昂贵的 **NVLink** 或 **InfiniBand** 网络，把这个 Token 的向量跨服务器“瞬移”到服务器 B。
- **底层坑点**：在写代码时，这一步叫 `torch.distributed.all_to_all`。如果你网络延迟高，或者 Token 发送的顺序乱了（没有做好 Padding 和对齐），你的分布式系统会直接卡死或算出乱码。

**V Cache 的噩梦再临**

- 我们在 Lecture 3 讲过 KV Cache 能加速推理。但在 MoE 里，KV Cache 变得更复杂了。 每个 Token 可能会跳跃到不同的专家，导致 KV Cache 在显存里的分布极其零碎。如果你没做好显存碎片整理，即便你总显存够用，也会因为无法分配出一块连续的空间而触发 **OOM (Out of Memory)**。
- 现代架构（如 DeepSeek）会配合我们之前讲过的 **GQA (分组查询注意力)**，极大压缩注意力部分的显存占用，把宝贵的空间全部腾出来给那几十个“吃内存”的专家。

## Upcycling 与 DeepSeek的一些启发

**Upcycling (向上平替/升级循环)**

- 从零开始训练一个 MoE 模型极其昂贵，且极易在初期崩溃。业界发明了一个极其取巧的捷径：拿一个已经训练好的稠密模型（Dense Model，比如 7B 参数），把它的 MLP 层**直接复制 N 份**，然后随机初始化一个 Router。 就这样，强行把它“伪装”成一个 MoE 模型接着练！
- 像 Qwen（通义千问）早期版本和 MiniCPM 都使用了这个方法，以极其低廉的算力成本，硬生生把旧时代小模型的智商拉高了一个维度。

**DeepSeek 的底层进化史 (从 V1 到 V3)**

- 其 MoE 核心（细粒度切割 + 共享专家）一直没变，但系统微操到了极致。
	- **V2 的跨节点限制 (Top M Devices)**：在分布式集群中，如果 Token 被路由发到了太多不同的服务器，网络通信带宽会直接卡死。V2 极其聪明地规定：先选定 **Top M 台服务器**，然后再在这些服务器内部挑专家。这从物理层面限制了跨节点的网络浪费。
	- **V3 的无损均衡补丁**：我们之前提过，V3 抛弃了传统的辅助损失函数，改用动态 Bias（偏置）做无损负载均衡。但为了防止在推理期遇到极端奇葩的提示词（Prompt）把某个专家单点卡死，V3 还是在底层偷偷加回了一个“序列级 (Sequence-wise)”的辅助损失作为保险。

**DeepSeek V3 非 MoE 黑科技 (MLA 与 MTP)**

- 这部分虽然不是 MoE，但绝对是支撑起 DeepSeek 霸榜的核心：
	- **MLA (Multi-head Latent Attention, 多头潜在注意力)**：上节课讲的 MQA/GQA 是强行减少 Key 和 Value 的头数来压缩 KV Cache。DeepSeek 不这么干！它引入了一个**低维潜在向量 $C$**。 它把高维的隐藏状态先“压缩”成极小的 $C$，并把 $C$ 存进显存缓存。等需要进行 Attention 运算时，再把 $C$ 瞬间解压缩（上投影）回 $K$ 和 $V$。配合矩阵乘法的结合律（把上投影矩阵直接融进注意力矩阵里），它在数学上完美消除了额外的计算开销，把推理期的显存占用降到了冰点。
	- **MTP (Multi-Token Prediction, 多 Token 预测)**：传统的自回归模型一次只猜下一个词。DeepSeek 在最顶层加了一个极其轻量级的小 Transformer 模块，让模型**一次性预测未来两个词**！ 这不仅加速了训练收敛，还能在推理时配合底层调度实现极速生成。

#  Lecture 5: GPUs

## GPU 物理结构与执行模型透视

**CPU 与 GPU 的哲学分歧 (Latency vs. Throughput)**

- **CPU (低延迟)**：就像是一辆顶级法拉利。它的硅片面积大部分给了“控制单元 (Control Unit)”和“分支预测”。它能用极高的时钟频率，极速跑完单条 C++ 线程。
- **GPU (高吞吐)**：就像是一列绿皮火车。它的硅片面积全是“算术逻辑单元 (ALU)”。它不管单个指令跑得多慢，它的目标是一次性拉运几万个矩阵乘法任务。

- **开发直觉**：在写 GPU 代码时，你必须彻底抛弃写 C/C++ 时的“单线程极速逻辑”，把思维切换到“万箭齐发”的阵列模式。

**GPU 物理黑盒解剖 (SM 与 SP)**

- 打开 GPU 的机箱，你会看到两个极其核心的物理层级：
	1. **SM (Streaming Multiprocessor, 流式多处理器)**：GPU 的“独立包工头”。一张 A100 显卡里大概有 108 个 SM。每一个 SM 都有自己独立的控制逻辑。在写 Triton 或 CUDA 代码时，你的核心调度单位就是 SM。
	2. **SP (Streaming Processor, 流处理器)**：在 SM 内部，塞满了无数个真正干活的数学计算单元。这些小单元负责疯狂地做浮点数乘加运算。

**内存墙与物理距离 (Memory Hierarchy)**

- 这是这节课最致命的考点！ 距离 SM 计算核心的物理距离，决定了你的程序是起飞还是坠毁。
	1. **Registers (寄存器) / L1 / Shared Memory (共享内存)**：就长在 SM 芯片**内部**。速度快到飞起（只需 20 个时钟周期）。
	2. **L2 Cache (二级缓存)**：长在 GPU 芯片上，但不在 SM 里，供所有 SM 共享。速度稍慢。
	3. **Global Memory / HBM (全局显存)**：**警告！它根本不在计算芯片上！** 它在 GPU 芯片的旁边。你要访问它，必须走排线通信。访问一次需要惊人的 200~300 个时钟周期！
- 如果你在代码里频繁申请全局数组（存放在 HBM），你的万级计算单元就会陷入漫长的等待（Idling）。优秀的底层代码，必须是一次性从 HBM 抠出一大块数据，搬到 Shared Memory 里疯狂循环计算，算完再送回去。

**GPU 的执行模型 (Blocks, Warps, Threads)**

- 硬件是 SM，那软件（代码）是怎么映射上去的呢？
	1. **Block (线程块)**：一组任务。调度器会把一个 Block 直接扔给一个 SM 去执行。
	2. **Thread (线程)**：最底层的单个计算任务（比如计算矩阵中的某一个像素）。
	3. **Warp (线程束) - 重点！**：GPU 嫌一个个调度线程太麻烦，它会把 **32 个线程** 死死绑定成一个 Warp。**这 32 个线程在任何一个时钟周期，必须执行极其相同的汇编指令 (SIMT: Single Instruction, Multiple Threads)！**

## 性能天花板与分支发散

**为什么只有“矩阵乘法”被奉为神明？**

- 深度学习之所以能爆发，全靠底层硬件的“偏心”。 如果你在 GPU 上做普通的加减乘除，速度其实一般。但如果你做**矩阵乘法 (MatMul)**，速度会瞬间飙升几个数量级！这是因为从 V100 架构开始，NVIDIA 在芯片里塞入了专门为了算矩阵而生的物理单元——**Tensor Cores (张量核心)**。 这也解释了为什么我们在前几节课看到的现代模型架构中，全都是极其纯粹的线性层 (Linear Layers)，连偏置项 (Bias) 都删了。因为在硬件层面，**除了矩阵乘法，其他的操作都不配占用宝贵的时钟周期！**

**越来越绝望的“内存墙”**

- 教授展示了一张极其残酷的摩尔定律对比图。 过去几年，GPU 的计算能力（算力）涨了十万倍以上（Super Exponential）！但是，GPU 全局显存（HBM）的读取带宽，仅仅涨了几十倍。 
- **物理结论**：现在的 GPU 就像是一个拥有恐怖马力的绞肉机，但上料的传送带极其缓慢。这就要求我们在设计算法时，必须抠抠搜搜地算计每一次访存（Memory Access），尽一切可能把数据留在片内的 Shared Memory 里反复利用。

**Roofline Model (屋顶线模型)**

- 当你发现你的模型跑得慢时，怎么判断瓶颈在哪？这时候就要祭出计算机体系结构里大名鼎鼎的 Roofline Model 了。 这张图的形状像一个房顶：
	1. **左侧斜坡 (Memory-Bound, 内存受限)**：算术强度（FLOPs/Byte）很低的操作（比如给矩阵加上一个常数，或者自回归生成下一个词）。此时你的极限性能完全受制于显存带宽，算力单元都在无聊地摸鱼。
	2. **右侧平顶 (Compute-Bound, 算力受限)**：算术强度极高的操作（比如两个巨大的方阵相乘）。此时数据传输已经完全满足了需求，真正限制你的是 Tensor Core 能算多快。

**条件分支与“分支发散” (Branch Divergence)**

- 编写 C/C++ 算法（比如备战蓝桥杯等编程竞赛时），写个 `if/else` 语句简直像呼吸一样自然，可以用来轻松处理不同的边界条件。但在 GPU 的微观世界里，这是绝对的禁忌！
	- **SIMT 的铁律**：上一节我们讲过，32 个线程被死死绑定在一个 Warp（线程束）里，它们**必须在同一时钟周期执行同一条指令**。
	- **惨剧发生**：如果你写了 `if (x < 0) { A } else { B }`，并且这个 Warp 里有一半的线程满足 `if`，另一半满足 `else`，会发生什么？ GPU 无法同时执行 A 和 B！它只能强行**暂停**一半的线程，先让一半人去执行 A；等 A 执行完，再把另一半人唤醒去执行 B。
	- **后果**：就加了一个小小的 `if/else`，你的系统吞吐量瞬间原地**腰斩 (Drop by 50%)**！

## 跨越内存墙的四大底层绝招

**降低精度 (Lower Precision & Mixed Precision)**

- 在 C++ 里，我们习惯用 4 字节的 `float` (FP32)。但如果你把 100GB 的 FP32 矩阵从显存搬到计算单元，通道直接就堵死了。

	如果换成 2 字节的 FP16 或 BF16，**数据搬运量瞬间砍半！** 在内存带宽不变的情况下，这等同于让你的程序速度直接翻倍！

- **混合精度 (Mixed Precision)**：当然，不能全用低精度，否则梯度会变成 0（下溢出）。聪明的做法是：输入数据用 16 位搬运，但在 Tensor Core 内部的累加器（Accumulator）里，用 32 位来保持高精度求和。

**算子融合 (Operator Fusion)**

- 假设你要算一段代码：`y = sin(x)^2 + cos(x)^2`。

	如果用最原始的 PyTorch（未优化），底层会发生什么极其愚蠢的事？

	1. 从显存读取 `x`，算 `sin(x)`，把结果写回显存。
	2. 再次读取显存，算平方，再写回。
	3. 这样来回折腾 5 次！这就像在工厂里，每做完一道小工序，就把半成品运回遥远的仓库，然后再运回车间。

- **Kernel Fusion (算子融合)** 就是手写一个巨大的 C++/CUDA 函数（Kernel），一次性把 `x` 读到极其快速的 Shared Memory（共享内存）里，一口气算完 `sin`、`cos`、平方和加法，最后只把最终结果写回显存。来回搬运次数从 5 次降到了 1 次！

	（这就是为什么你现在写 PyTorch 一定要加上一行神级代码：`torch.compile()`，它就是在底层自动帮你做算子融合！）

**重计算 (Recomputation / Activation Checkpointing)**

- 这是计算机科学里最经典的 **“以计算换空间 (Trade compute for memory)”**。

	前向传播时，正常的神经网络会把每一层的中间结果（Activations）都存到显存里，留着反向传播算梯度用。但这会吃掉海量的内存带宽！

	**绝招**：直接把这些中间结果丢掉（不写入显存）！等反向传播需要用到它们时，我们在算力极其富裕的计算单元里，根据输入 $x$ **当场重新算一遍**！虽然多废了一次算力，但省下了极其昂贵的内存读写时间，整体速度反而变快了。

**内存合并读取 (Memory Coalescing & Burst Reads)**

- 这可以说是 C++ 指针与底层硬件结合的最美妙瞬间。

	GPU 的显存硬件（DRAM）有一个物理特性叫 **“突发读取 (Burst Mode)”**：当你想读内存地址 `0` 的数据时，硬件其实是一次性把地址 `0, 1, 2, 3` 的一整块数据全抓过来给你。

- **Warp 与指针的对齐**：

	还记得那 32 个绑在一起的线程 (Warp) 吗？

	如果这 32 个线程，恰好去读内存中**连续相邻**的地址（比如线程 0 读 `A[0]`，线程 1 读 `A[1]`），硬件就会触发合并读取，一次物理 I/O 就能满足所有人！

	**反面教材**：如果你用 C++ 写了一个二维数组遍历，本来应该是按行（Row-major）读取，你非要用多线程去按列（Column-major）跨步读取。线程 0 读 `A[0]`，线程 1 读 `A[100]`……这会彻底打碎突发读取机制，导致 32 个线程触发 32 次缓慢的物理读取，内存吞吐量直接暴跌 90%！

## Tiling 分块与 Flash Attention

**Tiling (矩阵分块)**

- 做矩阵乘法 $M \times N$ 时，最愚蠢的 C++ 写法就是两个极其庞大的 `for` 循环嵌套。这会导致同一个矩阵元素被无数次从极慢的显存（Global Memory）里反复读取。

- **Tiling 的底层魔法**：

	既然 Shared Memory (共享内存/SRAM) 极快但容量极小，我们就把庞大的矩阵切成一块块小正方形（Tiles，比如 $128 \times 128$）。

	1. 让一个 Block 的线程，一次性把 $M$ 的一个 Tile 和 $N$ 的一个 Tile 搬进 Shared Memory。

	2. 在 Shared Memory 里疯狂执行乘加运算，算出这部分的所有局部和（Partial Sums）。

	3. 算完后，再去搬下一块 Tile。

		这相当于把去显存取数据的次数直接除了一个常数 $T$（Tile 的大小）。在硬件上，这为你挽回了数量级级别的性能！

**波浪量化 (Wave Quantization) 与维度对齐**

- **视频开头那个令人绝望的“锯齿状”性能图表终于破案了！为什么把矩阵维度随便加个 $1$，算力利用率会瞬间暴跌？

	- **物理原因 1 (内存对齐)**：如果你的矩阵维度不能被底层的“突发读取块”整除，线程在读取边缘数据时就会发生错位，导致内存读取次数翻倍。

	- **物理原因 2 (Wave Quantization)**：假设你的 A100 显卡有 $108$ 个 SM（流式多处理器）。经过 Tiling 切分后，你的任务恰好切出了 $120$ 个 Tiles。

		系统怎么调度？第一波，先派 $108$ 个 Tiles 把所有 SM 占满，完美 100% 运行。

		第一波跑完后，还剩 $12$ 个 Tiles。第二波，系统只能派这 $12$ 个 Tiles 去跑，**导致剩下的 96 个 SM 处于彻底空闲（Idling）状态！** 整体算力直接拉垮。

- 这就是为什么 OpenAI 的前大佬 Andrej Karpathy 宁愿往词表（Vocab）里强行塞入几十个无用的废 Token，也要把词表大小从 $5257$ 凑成 $5304$（因为它是 $64$ 的倍数）。凑齐 $2$ 的幂次，能让底层切块极其完美地填满 SM！

**Flash Attention**

- 普通的 Attention 最大的死穴在于，计算 $Q \times K^T$ 后，必须在显存里实例化一个庞大的 $N \times N$ 注意力得分矩阵。当上下文长度 $N$ 达到 $100k$ 时，单单这一个矩阵就能瞬间把显存撑爆（$O(N^2)$ 的内存复杂度）。

	我们既然学了 Tiling 和 Kernel Fusion（算子融合），能不能把 $Q, K, V$ 切成小块（Tiles），扔进 Shared Memory 里一口气算完，**根本不把 $N \times N$ 矩阵写回显存**？

- **最大拦路虎：Softmax**：

	Softmax 是一个“全局函数”。算分母时，你需要知道同一行**所有**元素的指数和。如果数据被切成了独立的 Tile，你在这个 Tile 里根本不知道别的 Tile 长什么样，怎么算全局 Softmax？

- **神级破局：Online Softmax (在线 Softmax)**：

	引入几个极其轻量级的寄存器变量：实时维护一个**“当前最大值 (Running Max)”\**和一个\**“当前分母和 (Running Denominator)”**。

	当我们处理下一个 Tile 时，如果发现了更大的值，就利用数学上的指数衰减特性（乘一个校正因子），极其优雅地动态修正前面已经算过的局部 Softmax 结果！

	就这样，Flash Attention 在 SRAM 内部通过流式处理（Streaming），极其完美地把 $O(N^2)$ 的内存读写降维成了 $O(N)$，彻底开启了大模型的长文本时代！

# Lecture 6: Kernels, Triton

## GPU 异步调度与 Benchmarking

**概念重载：从 C++ 到 GPU 的层级映射**

- 在写 Triton 或 CUDA 代码前，你的脑海里必须有一张极其清晰的硬件层级图：    1. **Grid (网格)**：你的整个总任务（比如一个巨大的矩阵乘法）。 2. **Block (线程块)**：任务被切碎后的基本调度单元。系统会把一个 Block 整体扔给一个 SM（流式多处理器）去执行。在一个 Block 内部，线程们可以极其快速地共享内存。 3. **Warp (线程束)**：32 个物理绑定在一起的线程，它们必须在同一时刻执行同一句汇编指令（SIMT）。 在接下来的代码里，你的思维必须始终在“我要开几个 Block？”和“这个 Block 里的偏移量 (Offset) 是多少？”之间来回切换。

**Benchmarking (基准测试) vs. Profiling (性能分析)**

- **Benchmarking**：算总账。用秒表掐一下，这行代码端到端跑完到底花了多少毫秒。它告诉你代码“慢不慢”。
- **Profiling**：做核磁共振。深入到 CUDA 底层，看看具体是哪个算子（Kernel）、哪次显存搬运耗尽了时钟周期。它告诉你代码“为什么慢”。 教授强调的铁律是：**在动手优化任何 C++/Triton 代码之前，必须先 Profiling！** 否则你可能会花 3 个小时去优化一个总耗时只占 1% 的无辜函数。

**异步执行与 `torch.cuda.synchronize()`**

- 这是无数深度学习新手（甚至老手）都会踩的巨坑！ 在 PyTorch 中，Python/C++ 代码运行在 CPU 上，而实际的矩阵乘法运行在 GPU 上。**这两者是极其割裂且异步 (Asynchronous) 的！**     当 CPU 执行到 `c = a @ b` 时，它只是把一个“计算指令”扔进 GPU 的任务队列里，然后 CPU 瞬间就往下一行代码跑了，根本不管 GPU 算没算完！ 如果你用普通的 `time.time()` 去测算，你测出的时间永远是零点几毫秒——因为你测的仅仅是 **CPU 发送指令的耗时**，GPU 那边可能还在苦哈哈地搬运显存呢。
- 在结束计时前，**必须**硬编码一行 `torch.cuda.synchronize()`。这行代码的底层逻辑是让 CPU 强行阻塞 (Block) 并挂起，死死盯着 GPU，直到 GPU 把队列里所有的 Kernel 全部执行完毕，CPU 才会继续往下走，这样你才能截取到真实的运行时间。

**Warm-up (预热迭代)**

- 在 GPU 上跑 Benchmarking，永远不要把第一次运行的时间算进去。 第一次调用 CUDA 算子时，底层会发生一堆繁杂的“行政工作”：初始化 CUDA Context、分配显存指针、甚至触发 JIT (即时编译) 把 Python 编译成底层机器码。这叫冷启动开销。必须先跑几次无意义的 Warm-up 让显卡“热起来”，再循环测算求平均值。

## PyTorch Profiler 与内核调度机制

**PyTorch 内置 Profiler**

- Benchmarking 只能告诉你“这段代码跑了 5 毫秒”，但这 5 毫秒里，CPU 和 GPU 分别在干嘛？

	调用 `torch.profiler`，它会监听程序运行时的每一个微秒，生成一份极其详尽的体检报告。它能精确列出：底层调用了哪个 C++ 函数、发射了哪个 CUDA Kernel（算子）、在 CPU 上耗时多少、在 GPU 上耗时多少。

**`Add` (向量加法)**

- 当你在 Python 里敲下 `a + b` 时，底层发生了什么？
	1. **Python 解释器**捕捉到运算符重载。
	2. 掉入底层 C++ 库，调用 `aten::add`（ATen 是 PyTorch 的核心 C++ 张量库）。
	3. C++ 准备好指针和维度信息，调用 `cudaLaunchKernel` 向 GPU 发送指令。
	4. GPU 收到指令，启动名为 `vectorized_elementwise_kernel` 的纯 CUDA 代码，并行执行加法。
- Profiler 显示，对于极其简单的加法，**CPU 准备和发射指令的时间，竟然比 GPU 实际做加法的时间还要长！** 这就是典型的“内核启动开销 (Kernel Launch Overhead)”。

**`MatMul` (矩阵乘法)**

- 当你调用 `a @ b`，Profiler 的画风变了。

	底层调用的不再是简单的 element-wise kernel，而是直接派发给了 **CUTLASS**（NVIDIA 官方极其变态的 C++ 模板库，专门用来把矩阵乘法压榨到硬件极限）。

	如果矩阵尺寸合适，你甚至能在 Profiler 里看到 `sm80_xmma...` 这样的字眼——这代表底层直接调用了汇编级指令，物理激活了 GPU 上的 **Tensor Cores (张量核心)**！

**复合操作的性能灾难 (`cdist`)**

- 假设你要算两个点集之间的欧氏距离，你调用了 `torch.cdist`。

	打开 Profiler 一看，灾难发生了！因为计算距离需要经历：相减 $\rightarrow$ 平方 $\rightarrow$ 求和 $\rightarrow$ 开根号。

	底层并没有一个专门叫“欧氏距离”的现成算子。于是，PyTorch 只能极其死板地把这个操作拆解：先调一次 `aten::mm`（乘法），再调一次 `aten::pow`（平方），最后调 `aten::sum`。

	**回忆 Lecture 5**：这触发了频繁的显存读写, 每一次独立的 Kernel 调用，都意味着 GPU 要把中间结果写回慢吞吞的全局显存，然后再读出来算下一步。如果我们在 C++ / Triton 里把它们写成一个**融合算子 (Fused Kernel)**，速度将产生质的飞跃！

## Nsight 深度诊断与手撕 CUDA C++ 算子

**Nsight Systems (nsys) 与 CPU/GPU 追逐战**

- 前面我们用 PyTorch 自带的 Profiler，只能看个大概。NVIDIA 官方的 **Nsight Systems** 才是底层的终极核磁共振！

	在图形化界面里，你会清晰地看到两条时间线：上方是 CPU，下方是 GPU。 **极其震撼的发现**：在正常训练时，CPU 的时间线往往**远远跑在 GPU 前面**！CPU 就像一个疯狂发牌的荷官，瞬间把未来 5 步的 CUDA Kernel 全扔进执行队列里，然后让 GPU 慢慢去算。

- 如果你在训练循环里写了一句 `print(loss.item())`，灾难就发生了！因为要打印具体的数值，CPU 必须硬生生停下来（触发隐式 Synchronize），死死干等到 GPU 算完那一步的 Loss 并把数据传回内存。这直接打断了 CPU 和 GPU 完美的异步流水线，让 GPU 算力瞬间出现真空期！

**从 Python GLU 到 C++ CUDA GLU**

- 我们以大模型里最火的 GLU 激活函数（及其变体 GELU）为例。
	- **Python 写法**：`0.5 * x * (1 + tanh(sqrt(2/pi) * (x + 0.044715 * x**3)))`。 这段代码在底层会启动加、乘、乘方、`tanh` 等四五个独立的 Kernel。测试耗时：**8.1 毫秒**。太慢了！
	- **PyTorch 原生写法**：调用内置的 `F.gelu`，耗时瞬间降到 **1.1 毫秒**！为什么？因为它底层是用 C++ 写好的 Fused Kernel。
- **手写 CUDA C++**： 现在，我们要用 C++ 语法，把上面那坨冗长的公式硬编码成一个纯粹的 CUDA 算子。 你会看到经典的 CUDA 函数签名：`__global__ void gelu_kernel(float* out, float* in, int num_elements)`。
	1. **黄金索引公式**：传进来的参数是指针 `in`，GPU 怎么知道当前这个线程该处理数组里的哪一个数字？ 你需要倒背如流这行 C++ 代码： `int i = blockIdx.x * blockDim.x + threadIdx.x;`*(逻辑：我在第几个线程块 $\times$ 每个块有多大 $+$ 我在块里的相对位置 = 我的全局绝对索引)*
	2. **边界保护 (Bounds Checking)**： 紧接着必须写 `if (i < num_elements)`。因为为了满足硬件对齐，我们启动的线程总数通常会稍微多于数组的实际长度。如果不加这句判断，极其暴力的越界指针访问（Segfault）会直接让你的 GPU 进程崩溃。
	3. **暴力计算**：在 `if` 内部，直接用 C++ 的数学库算完整个复杂的激活公式，然后 `out[i] = result;` 完美收工！

## 内存连续性防坑与 Triton 的优雅降临

**连续内存陷阱 (`is_contiguous`)**

- 在使用 C++ API 或者 Triton 发射 Kernel 之前，教授特意强调了一行极其关键的断言代码：`assert x.is_contiguous()`。**为什么？** 作为 C/C++ 开发者，你深知指针在内存中是线性移动的。在 PyTorch 中，如果你对一个矩阵做了转置操作 (`x.t()`)，底层的物理内存**根本没有发生搬运**，只是张量的跨度（Stride）元数据变了！ 如果你把一个非连续（Non-contiguous）的张量指针传给底层算子，你的 `x_ptr + offsets` 计算出来的物理地址将全是一堆垃圾乱码。在交给底层之前，必须强制它在内存里老老实实地连续排布！

**Triton 的 Block-Level 编程哲学**

- **写 CUDA 时**：你需要操心这 1024 个 Thread 谁负责读哪一个数字，还要绞尽脑汁考虑 32 个线程的 Warp 怎么对齐。
- **写 Triton 时**：你直接化身“包工头”。你的视角上升到了 **Block（线程块）** 级别！你不需要写 `for` 循环，而是直接生成一个巨大的向量数组。Triton 编译器（隐藏在底层的超级大脑）会自动帮你把这个庞大的向量操作拆解成单线程指令，并极其聪明地帮你把数据搬进 Shared Memory。

**手撕 Triton GLU 算子 (Offsets 与 Mask)**

- **C++ 指针逻辑的 Python 化**：
	1. **指针偏移 (Offsets)**： 在 Triton 里，我们不再算单点索引，而是算一整个 Block 的索引数组： `offsets = block_start + tl.arange(0, BLOCK_SIZE)` 这就好比直接在 C++ 里抓取了一个长度为 `BLOCK_SIZE` 的数组引用。
	2. **越界掩码 (Masking)**： 由于我们的任务总是被切分成一个个饱满的 Block，遇到数组尾巴时，极其容易出现越界（Out of Bounds）。 在 Triton 里绝对不要写 `if` 语句（避免分支发散），而是写： `mask = offsets < num_elements` 在执行物理内存读取时，直接挂上掩码：`tl.load(x_ptr + offsets, mask=mask)`。Triton 底层会自动把越界部分的内存读取屏蔽掉，极其安全优雅！

**剖析 PTX 汇编代码**

- Triton 编译出来的 PTX 代码（GPU 的底层汇编指令）能看到一行底层的机器指令：`LD.global`。 更精妙的是，底层硬件**一次性极其贪婪地读取了 4 个寄存器的值**！这完美印证了我们在 Lecture 5 学过的硬件物理法则：**突发读取 (Burst Reads) 和 内存合并 (Memory Coalescing)**。Triton 编译器没骗人，它在底层帮你把访存优化压榨到了极限！

## **JIT 编译与手撕 Triton Softmax 终极挑战**

**`torch.compile` (JIT 编译器)**

- 我们之前花大力气用 C++ 和 Triton 写的 GLU 融合算子，如果直接给原生的 PyTorch 代码套上一个 `torch.compile()`，运行时间竟然直接降到了 **1.3 毫秒**！比我们手写的还要快一点点！
- **C++ 底层映射**：`torch.compile` 就像是 C++ 编译器里的 `-O3` 和 `-march=native` 优化开关。它会在底层进行图捕捉 (Graph Capture)，自动把你能想到的那些无脑 `for` 循环和元素级计算 (Element-wise) 融合成一个极其高效的 Triton Kernel。
- **边界**：像 Flash Attention 这种需要切块 (Tiling)、重计算 (Recomputation)、精细控制 Shared Memory 驻留时间的算法，JIT 编译器目前根本推导不出来。遇到这种核心算子，还是得靠我们手写。

**Triton Softmax**

- 前面的 GLU 是“元素级 (Element-wise)”操作，各个线程互不干扰。但 Softmax 是一个**规约操作 (Reduction)**！

	公式是：$y_i = \frac{e^{x_i}}{\sum e^{x_j}}$。你要算出一个元素的值，必须知道**这一整行**的总和！

- **Triton 的高维解法**：

	1. **网格设计 (Grid)**：既然要算一整行，最优雅的做法是**让一个 Block (线程块) 强行吃下整个一整行！** 这样网格 (Grid) 就是一维的（大小为矩阵的行数）。
	2. **内存对齐 (Padding)**：显存读取必须对齐！如果一行的长度是 $1000$，在 Triton 里我们必须把它强行拉伸到最近的 $2$ 的幂次方（`triton.next_power_of_2(N)`），即 $1024$。多出来的那 $24$ 个位置用掩码 (Mask) 盖住。
	3. **防爆处理 (Numerical Stability)**：在求指数之前，必须先找到这一行的最大值，所有元素先减去这个最大值（`row = row - tl.max(row, axis=0)`），然后再进行 `tl.exp()` 和 `tl.sum()`。这在底层 C++ 中是极其标准的防浮点数溢出操作。

- **性能碾压**：如果我们用纯 Python 循环写 Softmax，底层会触发 `max`, `sub`, `exp`, `sum`, `div` 五个独立的 Kernel，内存带宽当场爆炸。而我们用 Triton 写的，是一个极其完美的单一 Fused Kernel，所有数据在 SRAM 里一次性完成规约，速度碾压！

## **集群通信墙与 Ring All-Reduce 魔法**

**组建集群(The Compute & Memory Wall)**

- 一张 H100 只有 80GB 显存。之前我们算过显存四大巨兽（模型权重、激活值、梯度、优化器状态）。如果是全参数训练一个几十亿参数的模型，加上庞大的 Batch Size，单卡在第一步就会直接 OOM（爆显存）。更别提算力了，单卡跑完 GPT-3 需要几十年的时间。
- **物理拓扑**：所以我们必须加卡！在服务器内部，8 张 GPU 用极高速的 **NVLink** 连接；而在服务器和服务器之间，必须用高昂的 **InfiniBand (IB) 网络** 组成物理集群。

**数据并行 (Data Parallelism, DP)**

- 既然一张卡算得太慢，最直觉的分布式方法是什么？
	1. **复制模型**：把一模一样的模型权重，完整地复制到 8 张卡上。
	2. **切分数据**：如果当前的 Batch 有 800 个 Token，我们把它切成 8 份，每张卡只负责算 100 个 Token 的前向传播和反向传播。
	3. **梯度合并**：每张卡算完后，手里都拿着一份专属于自己那 100 个 Token 的梯度矩阵 `dW`。
- **致命危机**：如果这时候让 8 张卡各奔东西，各自用自己的 `dW` 去更新参数，那这 8 张卡里的模型就会长得完全不一样，整个大模型当场精神分裂！**所以，在更新参数前，我们必须把这 8 张卡上的 `dW` 全部加起来求个平均，然后再分发回每张卡。** 这就是分布式的核心难点：**同步 (Synchronization)**。

**参数服务器 (Parameter Server) 的崩溃**

- **C++ / 网络并发透视**：在早期的深度学习里，大家用的是 Master-Worker 架构。

	指定 GPU 0 当“老板”，GPU 1-7 当“打工人”。算完梯度后，GPU 1-7 同时把自己几 GB 的梯度矩阵通过网络发给 GPU 0。

	**灾难发生了**：这在网络编程里叫极其严重的**扇入 (Fan-in) 拥塞**！GPU 0 的网卡瞬间被 7 路庞大的数据流挤爆，而其他 7 张卡在发完数据后只能原地干等（Idling）。系统的木桶效应极度明显，哪怕你加到 100 张卡，训练速度也根本不会变快，全堵在中心节点的网卡上了。

**环形全规约 (Ring All-Reduce)**

- 为了打破中心节点的拥塞，百度和 NVIDIA 的架构师们引入了高性能计算（HPC）领域极其优美的拓扑算法：**Ring All-Reduce**。

	1. **取消中心节点**：让 8 张卡首尾相连，组成一个物理圆环。
	2. **矩阵切块 (Chunking)**：把庞大的梯度矩阵 `dW` 均匀切成 8 个小块。
	3. **接力传递 (Scatter-Reduce & All-Gather)**：GPU 0 把第 0 块发给 GPU 1 的同时，GPU 1 把第 1 块发给 GPU 2……大家同时向右边传递一个极小的切块，并在收到左边传来的切块后执行一次原地的 C++ 数组加法。

- **底层数学之美**：经过 $2 \times (N-1)$ 轮轻量级的环形传递，所有 GPU 的显存里都神奇地拥有了完全一模一样的、全网平均后的梯度矩阵！

	最无敌的是：在这个环形拓扑里，**每一张网卡的上传和下载带宽都被 100% 完美利用**，彻底消灭了拥塞！它的通信耗时，在数学上几乎不随卡数的增加而增加。

## **ZeRO 与 FSDP 的内存魔法)**

**传统 DP 的显存大浪费 (Memory Redundancy)**

- 在传统数据并行 (DP) 中，如果你有 8 张卡，为了让每张卡能独立算前向和反向传播，这 8 张卡在显存里都必须老老实实地存下一份**极其完整**的 70B 模型权重、极其完整的梯度 `dW`，以及极其完整的 Adam 优化器状态！

**ZeRO (零冗余优化器) 的三段进化**

- 微软的架构师们提出了极其暴力的解法：既然有 8 张卡，那我们就把所有需要存的东西**切成 8 份 (Sharding)**，每个人只保管自己的那 $\frac{1}{8}$！
	- **ZeRO-1**：只切分“优化器状态 (Optimizer States)”。（立马砍掉一大半显存，因为 Adam 的一阶和二阶动量极其吃内存）。
	- **ZeRO-2**：不仅切优化器，连“梯度 (Gradients)”也切成 8 份！
	- **ZeRO-3**：最丧心病狂的一步，连“模型权重 (Parameters)”本身都切成 8 份！

**FSDP (完全切片数据并行) 的底层运转**

- **ZeRO-3 在 PyTorch 里的官方实现就是 **FSDP (Fully Sharded Data Parallel)**。

	如果 GPU 0 只有模型的第 1 层权重，它算到第 2 层时没数据了怎么办？

- **FSDP 的“拿来即用，用完即丢”魔法**：

	1. **All-Gather (全网收集)**：当 GPU 0 准备算第 2 层时，它立刻通过 NVLink/IB 网络大喊一声，把 GPU 1-7 手里保管的第 2 层权重碎片瞬间“收集”过来，在自己的 VRAM 里拼凑出一个完整的第 2 层。
	2. **Compute (计算)**：执行极其快速的矩阵乘法。
	3. **Free (释放内存)**：算完这层后，GPU 0 **毫不犹豫地立刻释放掉 (Deallocate/Free)** 刚才收集来的别人的权重，把显存清空，准备去收集第 3 层！

- **架构代价**：这种做法极其疯狂地增加了网络通信量（大约是原来的 1.5 倍），但换来的是**单卡显存占用缩小了 $N$ 倍（$N$ 是集群的 GPU 数量）**！

## **极致切分：张量并行与流水线并行的黑魔法**：**

**张量并行 (Tensor Parallelism, TP) —— 矩阵的物理撕裂**

- 假设你的 MLP 层有一个极其庞大的隐藏层矩阵，连 FSDP 都救不了。

	TP 的做法极其残暴：直接把这个矩阵“劈开”！

	在线性层 $y = x A B$ 中：

	1. **列切分 (Column Parallel)**：把矩阵 $A$ 按列切成 $A_1$ 和 $A_2$。GPU 0 算 $x A_1$，GPU 1 算 $x A_2$。此时两张卡互相独立，不需要通信。
	2. **行切分 (Row Parallel)**：紧接着过矩阵 $B$ 时，把 $B$ 按行切成 $B_1$ 和 $B_2$。GPU 0 继续算第一部分，GPU 1 算第二部分。
	3. **瞬间汇合 (All-Reduce)**：算完之后，两张卡必须立刻调用 `All-Reduce`，把局部的加和结果合并成最终结果，才能进入下一层！

- **极端的硬件要求 (NVLink 独占)**：

	作为底层开发者，你肯定立刻发现了 TP 的死穴——它在**一个前向传播的内部**发生了极其密集的网络通信！如果这步通信走的是普通的网线，延迟会瞬间把 GPU 卡死。所以，**张量并行 (TP) 绝对不能跨服务器！它只能在同一台机器的 8 张显卡内部，通过带宽极高的 NVLink 物理总线来执行。**

**流水线并行 (Pipeline Parallelism, PP) —— 纵向切割**

- 既然单机 8 卡被 TP 占满了，如果模型还有几十层装不下，怎么跨服务器扩展？答案是**流水线并行 (PP)**。

	把模型纵向切断！GPU 0 负责第 1 到 10 层，GPU 1 负责第 11 到 20 层……

- **致命缺陷：流水线气泡 (Pipeline Bubble)**：

	如果是写普通 C++ 多线程，你会发现一个灾难：当 GPU 0 在算第 1 到 10 层时，GPU 1 到 7 都在原地干等（Idling）！等 GPU 0 算完把数据传给 GPU 1，GPU 0 又闲下来了。这在系统监控图上会出现大片空白，被称为“气泡”。

**微批次流水线 (Micro-batching)**

- 为了填满气泡，谷歌 (GPipe) 和微软 (PipeDream) 发明了微批次调度。

	不要一次性把 800 个 Token 的 Batch 喂进去。把它切成 8 个大小为 100 的**微批次 (Micro-batches)**！

	GPU 0 算完第一个微批次，立马把它扔给 GPU 1，然后 GPU 0 **瞬间无缝衔接**开始算第二个微批次。这样就像工厂的流水线一样，除了首尾有一点点极小的气泡启动时间，中间所有的 GPU 都在满负荷疯狂运转！

## **大结局：3D 并行拓扑与长文本切片)**

**3D 并行拓扑映射 (3D Parallelism)**

- 假设你有 1024 张 GPU，你怎么分配 TP, PP 和 DP？

	这极其考验分布式系统的物理直觉：

	1. **张量并行 (TP=8)**：因为 TP 的通信极其密集（每次矩阵乘法都要 All-Reduce），它**绝对不能**跨越极其缓慢的网线！我们把它死死锁在单台服务器的 8 张显卡内部，独享极速的 NVLink 物理总线。
	2. **流水线并行 (PP=8)**：模型太深了，单机装不下。我们把 8 台服务器连起来，形成一条流水线。GPU 通信只发生在相邻的两层之间（点对点通信 Point-to-Point），对网络带宽要求最低。
	3. **数据并行 (DP=16)**：现在我们有了一个由 64 张卡组成的一个“完整模型副本”。剩下还有无数张卡怎么办？直接复制这个大副本 16 份，在它们之间做我们最熟悉的 Ring All-Reduce 梯度同步！

**流水线气泡的数学推导 (Pipeline Bubble Math)**

- 我们上个切片讲过，微批次 (Micro-batch) 能填补流水线气泡，但到底能填补多少？

	底层架构师有一个极其经典的计算公式：

	气泡浪费的时间比例 $\approx \frac{P-1}{M}$ （其中 $P$ 是流水线划分的层数级，$M$ 是微批次的数量）。

- 为了让浪费的时间趋近于 $0$，你的微批次 $M$ 必须**远远大于**流水线级数 $P$。但代价是，你的 GPU 显存里要同时驻留 $M$ 个批次的激活值 (Activations)，极其容易 OOM！这又是一场 C++ 内存管理与系统吞吐量之间的极限拉扯。

**序列并行 (Sequence Parallelism, SP / Ring Attention)**

- 大模型时代，大家都想做 1M (一百万) 长度的超长上下文。

	但是，TP 只切矩阵维度，DP 只切 Batch Size。如果你的 Batch Size 是 1，但 Sequence Length 是 1,000,000，传统的 3D 并行全部失效！GPU 照样被单句长文本撑爆显存。

- 把长句子切碎！GPU 0 负责第 1 到 10万 个词，GPU 1 负责 10万 到 20万 个词。

	在算 Attention 的时候，利用我们第一节课讲的 **Ring All-Reduce 环形思想**，让 GPU 们在环形网络里互相传递 KV Cache 碎片！GPU 0 算完自己那块后，把 KV 发给左边的兄弟，同时接收右边兄弟传来的 KV 继续算。完美消灭了长文本的内存瓶颈！

### GPU 集群的物理硬件拓扑与通信鸿沟

**它在讲什么？** 揭示了在多节点、多任务的真实训练集群中，数据是如何在不同的存储层级（从极其快速但容量极小的 L1 缓存，到 GPU 的主显存，再到跨越物理主板的网络）之间流动的。

**为什么会有这个东西？** 在千亿参数大模型的训练中，真正的瓶颈往往不是浮点运算能力，而是**内存墙 (Memory Wall)** 和**通信墙 (Communication Wall)**。当我们把巨大的模型张量切分到多张物理显卡上时，必须进行高频的内存同步。传统的计算架构采用 PCIe 总线，数据需要进入操作系统的内核态，通过 CPU 转发，甚至走以太网 TCP/IP 跨机器传输，这引入了极大的系统开销。必须引入专为数据流式计算而生的物理链路。

**更正式一点的理解**

- **高带宽内存 (HBM, High Bandwidth Memory)**：集成在 GPU 上的“主存”，带宽极高，但容量相比普通内存要小得多。
- **PCIe 总线**：传统架构下的数据主干道。虽然通用，但它是 GPU 访问宿主机 CPU 和硬盘的主要通道，带宽十分受限。
- **NVLink**：Nvidia 专门为 GPU 集群互联设计的专用物理总线，直接将多块物理 GPU 桥接成一个对等的计算网格，彻底绕过 CPU 的内存池和操作系统的调度机制。在现代的 H100 集群上，NVLink 的单卡聚合带宽可以达到惊人的 900 GB/s。
- **NVSwitch**：一种专门用于在服务器机架之间，甚至跨越整个机房互联 GPU 的大型网络交换机架构，其设计目标就是用最低的延迟取代传统数据中心的以太网 (Ethernet) 协议栈。

**工程/实现视角** 在编写底层的 C/C++ 或 CUDA 并发代码时，如果不严格把控物理拓扑，极容易写出跨越节点的高频读写循环（例如在一个大 `for` 循环中反复从另一台机器拉取一小块显存）。在这种内存布局失误下，你的程序瓶颈会 100% 卡在网卡阻塞和系统调用开销上，算力利用率 (MFU) 会暴跌至个位数。

**一个小例子 / 小推演** 设想在同一台服务器上有两块物理 GPU。如果它们通过常规的 PCIe 互相读取对方的数组，数据包必须先拷贝进系统内核缓冲区，唤醒 CPU，再通过另一条 PCIe 通道转发，不仅延迟高，而且严重挤占了宿主机的系统带宽。而有了 NVLink 后，底层的 DMA（直接内存访问）可以直接从对端 GPU 的内存地址读取数据，代码上就像跨线程访问本地连续指针 `float*` 一样无缝且极速。

**容易误解的点**

- 不要误以为只要调用了多进程并行，系统底层的网络传输就绝对快。如果不配置针对性的后端库（如 NCCL），底层仍然可能会回退到缓慢的传统网络协议栈。
- 更准确地说，虽然 NVLink 的传输速度极快，但它的延迟和带宽瓶颈依然远低于访问同一块硅片上的 L1/L2 缓存。

## 集合通信原语 (Collective Operations) 与 NCCL

定义了一套标准的宏观网络操作接口，允许系统中的所有并发参与者（无论是一台机器内的多张显卡，还是跨数据中心的多台服务器）以最高效的数学约定交换、汇总和覆盖各自内存里的张量数据。

在大规模并发 C/C++ 系统中，如果要求开发者手动调用底层的 Socket 接口去管理数十个节点之间点对点 (Point-to-Point, 发送/接收) 的状态机，业务逻辑将变得极其混乱。一旦出现微小的时序错乱，网络死锁和缓冲区拥塞几乎是致命的。这套原语彻底封装了底层路由细节，解决了“如何安全、无锁、优雅地让全网节点互相交换数据”的工程难题。

集合通信非常像一场跨部门的高效会议规范：

- 领导单向宣读文件，所有人静听，这就叫 **Broadcast（广播）**。
- 主管把一个大任务切碎，分别私发给不同的人员去执行，这就叫 **Scatter（散播）**。
- 所有人做完手头的零件后，全交还给主管统一组装，这就叫 **Gather（收集）**。
- 所有人把工作进度在一个共享大屏上透明地拼合起来，所有人都能同时看到全局拼图，这就叫 **All-Gather（全收集）**。

要驾驭分布式原语，必须将两个关键的系统环境变量刻入脑海：

- **全局规模 (World Size)**：当前任务中参与并发通信的独立进程或显卡总数（例如 4 张卡同时跑，World Size 就是 4）。

- **进程标识 (Rank)**：每个独立运行设备的身份索引号，从 $0$ 一直排到 $\text{World Size} - 1$。

	核心的四种复合内存操作：

1. **Reduce（规约）**：将所有 Rank 对应内存块里的张量通过某个可交换的操作（如按位求和 Sum、求极值 Max）合并，并将最终结果写入某个特定的目标 Rank 中，其他 Rank 不保留结果。
2. **All-Reduce（全规约）**：执行规约计算，但是它会把最终聚合出来的完美结果毫无保留地覆盖回**所有 Rank** 的内存里。
3. **Reduce-Scatter（规约-散播）**：首先对全局进行求和或聚合，但随后立即把结果张量**切分成不重叠的等长块**，按照索引将每一块分发给对应的 Rank。
4. **All-Gather（全收集）**：每个 Rank 起初只保管一小块局部的张量切片。原语执行后，底层会将所有人的切片像搭积木一样串联起来，广播给所有 Rank，每个人都获得了一个极其庞大的完整数组拷贝。

**工程/实现视角**

- **集合通信库 (NCCL, Nvidia Collective Communication Library)**：Nvidia 官方提供的底层 C++ 通信引擎。它最强悍的地方在于，它会智能探测硬件拓扑，在底层构建最优的环形 (Ring) 或者是树状数据接力路径。
- PyTorch 原生的 `torch.distributed` 模块本质上就是给底层的 NCCL（针对 GPU 集群）或 Gloo（针对 CPU 后台测试）加了一层 Python 的动态绑定映射。
- **同步屏障 (Barrier, `dist.barrier()`)**：这是一个能救命的 API。因为系统里成百上千个 Rank 都是完全无感、异步执行的 C/C++ 进程，有些节点运算快，有些慢。在进入可能导致死锁的关键网络交换区前，必须插入一句 Barrier，这就相当于一把刚性的全网阻塞锁，强迫跑得快的人停下来挂起，直到最慢的 Rank 也抵达该行代码后，大家再统一步调进入下一阶段。

## 通信带宽基准测试 (Benchmarking) 与 All-Reduce 的“乘 2 惩罚”

这一段教我们如何通过掐秒表（计算耗时）和计算传输字节数，来测量一段分布式代码到底占用了多少物理网络带宽。

硬件厂商（比如 Nvidia）在宣传册上写着 NVLink 的峰值带宽是 900 GB/s，但如果在写 Python 分布式训练脚本时盲目相信这个数字，你的性能预估会错得离谱。实际运行中会有系统调用开销、协议开销和硬件调度损耗。为了精准定位大型集群的性能瓶颈，我们必须亲手编写测试脚本，测出实际代码跑出来的**有效带宽 (Effective Bandwidth)**。这种极致的 Profiling 思维，是进入高阶系统研究的必备素质。

这就像测试你家宽带的极限网速。你不能只看路由器包装盒上的参数，必须亲手拉取一个巨大的文件，用秒表算出总耗时，然后用 $\frac{\text{文件大小}}{\text{总耗时}}$ 来得出真实网速。只不过在分布式深度学习里，这个“大文件”是一个拥有上亿参数的张量 (Tensor)。

带块吞吐量的基础计算公式非常直白：

$$\text{Bandwidth} = \frac{\text{Bytes\_Sent}}{\text{Duration}}$$

当你调用一次 `All-Reduce` 让所有节点的显卡汇总梯度时，底层发生了什么？

从单一 Rank（某一张显卡）的视角来看，它经历了两个阶段的物理流量：

1. 它必须把自己的张量切片**发送出去**，去参与全局的汇总（对应 $\times 1$ 的通信量）。

2. 全局汇总完成后，它还要把那个完整的、最终合并好的张量从网络上**拉取回来**并写入自己的显存（对应再加 $\times 1$ 的通信量）。

	因此，对于 `All-Reduce` 来说，通信量存在一个理论上的倍数关系：

	$$\text{Bytes\_Sent} = 2 \times \text{Tensor\_Size\_in\_Bytes}$$

在使用类似于 C/C++ 底层思维与内存分配的操作时，测量时间的逻辑极其讲究。在调用计时的 `start_clock` 之前，必须先跑一次无用的空函数（Warm-up），并调用 `dist.barrier()` 和显卡流同步操作。如果不这么做，你测出的时间会包含 GPU 内核启动 (Kernel Launch) 的初始化延迟、操作系统分配连续内存指针的耗时，这会导致算出来的带宽小得可怜。这就叫排除了冷启动污染的“纯净 Benchmark”。

**一个小例子 **

假设我们用 PyTorch 创建了一个包含 1 亿个元素的 `float32` 张量：

- 张量大小计算：$10^8 \times 4 \text{ Bytes} \approx 400 \text{ MB}$。
- 执行一次 `All-Reduce`，该节点实际在物理互联上吞吐的数据量是 $2 \times 400 \text{ MB} = 800 \text{ MB}$。
- 如果掐表测得总耗时为 0.0028 秒。
- 实际测得带宽就是 $800 \text{ MB} / 0.0028 \text{ s} \approx 285 \text{ GB/s}$。这离理论峰值 900 GB/s 差了很远，说明硬件层或驱动层存在不可忽视的损耗。

**容易误解的点**

- **不要误以为**单次 API 调用的耗时仅仅是纯粹的数据传输时间，它还包含了底层算子的寻址与同步挂起开销。
- **更准确地说**，乘 2 惩罚（Factor of 2）是一个用于估算系统理论极限的经典宏观法则。实际的底层拓扑（如 NCCL Ring 算法）会把数据切得更碎来榨干上下行全双工信道，但宏观上通信量的总量预估依然服从这个 $\times 2$ 定律。

## Reduce-Scatter 的测量与组合直觉印证

使用相同的测试手段，对 `Reduce-Scatter`（规约-散播）这个原语进行带宽基准测试，并从物理数据量的角度印证它和 `All-Reduce` 之间的数学联系。

因为 `All-Reduce` 那极其沉重的 $\times 2$ 流量惩罚在超大规模集群中是致命的。为了极致压缩显存（例如后面的 FSDP 全切片数据并行），工程师们会极其克制地只使用单向的通信操作，因此需要精准确认 `Reduce-Scatter` 到底能省下多少网络带宽。

如果说 `All-Reduce` 是“所有人上交考卷 + 批改 + 所有人领回全班标准答案”（双程票）；那么 `Reduce-Scatter` 就仅仅是“所有人上交卷子，而且每个人只负责批改其中的一小段”（单程票）。既然只有单程，理所当然通信量就少了一半。

在 `Reduce-Scatter` 的测试中，因为最终的结果是直接被打散（Scatter）存储在各个 Rank 的碎片内存区域里，不存在最后一步“广播完整答案”的动作，所以它的总通信量**没有**乘以 2 的系数：

$$\text{Bytes\_Sent} = \text{Tensor\_Size\_in\_Bytes}$$

在这个视频演示中，讲师运行代码后发现，虽然理论上 `Reduce-Scatter` 通信量减半，但它测出来的带宽表现甚至更低（只有大概 70 GB/s）。这就向我们展现了底层系统的一个残酷现实：理论算法的高效，并不总是等于物理执行的高效。可能硬件厂商在硅片内部对最高频的 `All-Reduce` 做了特异性的微指令物理加速，而对 `Reduce-Scatter` 的优化则没有那么极致。

## 数据并行 (DDP) 的代码级手撕



数据并行 (Distributed Data Parallel, DDP) 是最直观的并行方式。模型保持完整不切割，每个 GPU 上都有一份 100% 完全相同的模型参数克隆体。我们唯一切割的是输入的数据集（即 Batch 维度）。

- **旧方法的问题**：如果单纯在一张卡上用极大的 Batch Size 训练，激活值 (Activations) 会瞬间把显存撑爆（OOM）。
- **新方法 (DDP) 怎么改**：把一个巨大的 Batch 切碎，分发给多张卡同时算。算完后，大家把各自算出的“梯度”加起来平均一下，然后再各自更新自己的参数。这样既扩大了整体的吞吐量，又不会撑爆单卡的显存。
- ![Data Parallelism in Deep Learning，AI 生成](https://encrypted-tbn0.gstatic.com/licensed-image?q=tbn:ANd9GcS1V0zOQqrdKorQIJaMZEpZ_UptLPPSKTGfi2b0BUnhZASN3Cz8sTUh9iYmkk8zfIpusn-ukBV-nRbr4KDqwrycurvWnq234LoBBrqEOy4prdGeBzc)Getty Images

就像是一场有 128 道题目的期末考试，需要 4 个助教（GPU）来批改。为了提速，每个助教只分到 32 道题（切分 Batch）。每个人手里都拿着一模一样的“标准答案评分细则”（模型权重）。批改完之后，4 个助教必须开个会，把他们发现的“题目扣分规律”（梯度）汇总平均一下，然后大家统一用这个平均规律去修改明年的评分细则（更新参数）。

在代码层面，我们需要通过 `world_size` 和 `rank` 来控制内存指针的偏移：

- **数据切片**：

	$$\text{local\_batch\_size} = \frac{\text{batch\_size}}{\text{world\_size}}$$

	Rank 为 $i$ 的设备，去主内存里抓取索引从 $i \times \text{local\_batch\_size}$ 开始的数据块。

- **前向与反向传播**：和普通的单卡 SGD 没有任何区别。每个 Rank 各自算出自己的 `loss` 和局部的梯度 `param.grad`。

- **神来之笔 (梯度同步)**：在反向传播 `loss.backward()` 之后，优化器 `optimizer.step()` 之前，插入一句极其关键的代码：

	对每一层的梯度执行 `All-Reduce(mean)`。

	$$g_{global} = \frac{1}{\text{world\_size}} \sum_{i=0}^{\text{world\_size}-1} g_i$$

DDP 的代码实现是非侵入式的（Non-intrusive）。你完全不需要去修改模型（比如 MLP 或 Transformer）内部的前向传播数学公式。你只是像一个黑客一样，拦截了普通 SGD 循环中的梯度变量，强行做了一次网络通信（`All-Reduce`），然后再放行。

## 张量并行 (TP) 的代码级手撕

张量并行 (Tensor Parallelism, TP) 是一种模型并行。这次我们不切数据了（大家吃一样的数据），我们要把组成神经网络的极其庞大的**矩阵乘法**，在维度上直接切碎，分给不同的显卡算。

- **旧方法的问题**：当你的参数规模来到 70B 或千亿级别，哪怕是 Batch Size = 1，单张 80GB 的 H100 也根本塞不下模型本身的权重和优化器状态。DDP 在这里直接报废。
- **新方法 (TP) 怎么改**：把那个塞不下的超大权重矩阵拦腰斩断（或者垂直切开）。每个 GPU 只在显存里保存这个矩阵的 $\frac{1}{N}$，并在前向传播时，只负责计算属于自己的那一小部分矩阵乘积。

这次不是批改 128 道题了，而是只有 1 道极其变态、写满一整面黑板的超级微积分大题（一次巨大的矩阵乘法）。4 个助教一起看着这 1 道题（同一份 Batch 数据）。助教 A 算矩阵的上半部分，助教 B 算下半部分。算完之后，如果不把 A 和 B 的结果拼起来，下一道题就没法做了。所以，每次算完一步，他们必须在黑板上把各自的答案缝合展示出来（通信拼图）。

以一个隐藏层维度为 `num_dim` 的 MLP 线性层为例：

- **权重切片**：

	$$\text{local\_num\_dim} = \frac{\text{num\_dim}}{\text{world\_size}}$$

	原本形状为 $(D, D)$ 的参数矩阵，在显存中被切成了形状为 $(D, \text{local\_num\_dim})$ 的长条。每个 GPU 只加载这一个长条。

- **前向乘法 (局部计算)**：

	输入 $X$ 的形状是 $(B, D)$。当前 GPU 上的权重 $W_{local}$ 形状是 $(D, D/\text{world\_size})$。

	两者相乘，得到的局部激活值 $A_{local}$ 形状是 $(B, D/\text{world\_size})$。

- **张量缝合 (通信操作)**：

	下一层网络期望的输入形状必须是完整的 $(B, D)$。因此，在这一层算完后，必须立即调用一次网络通信：

	调用 **`All-Gather`** 原语，把 4 张卡上形状为 $(B, D/4)$ 的结果沿着列方向拼接起来，还原出 $(B, D)$ 的完整激活值，再喂给下一层。

## 流水线并行 (PP) 与微批次调度

流水线并行是按照神经网络的**层数 (Layers)** 进行物理切分。如果模型有 4 层，Rank 0 负责计算第 1、2 层，计算完的激活值通过网络丢给 Rank 1；Rank 1 接力计算第 3、4 层。

如果只用张量并行 (TP)，我们不仅要在单台机器内部频繁通信，当模型参数达到几百 B 甚至万亿规模时，单台拥有 8 张 GPU 的服务器根本装不下哪怕一层的矩阵切片。如果让 TP 跨越服务器的以太网通信，极低的带宽会瞬间卡死整个训练。把极深的模型“按层”切断，放在不同的服务器上。因为网络层与层之间的激活值 (Activations) 数据量相对较小，跨越服务器甚至跨机架传输时，对网络带宽的要求远低于 TP，非常适合跨节点的高延迟网络。

这就好比工厂的流水线。GPU 0 是第一道工序（打磨零件），GPU 1 是第二道工序（喷漆）。如果 GPU 0 一次性把 800 个零件全打磨完才交给 GPU 1，那么在 GPU 0 疯狂打磨的这段时间，GPU 1 只能干瞪眼处于空载状态（这就是**气泡**）。为了解决这个问题，我们采用**微批次 (Micro-batching)**：GPU 0 打磨好 100 个零件，立马递给 GPU 1 喷漆，然后自己无缝回头去打磨下 100 个。这样两台机器就能同时运转了。

- **数据结构转换**： 设全局批次大小为 `batch_size`。为了掩盖传输延迟，我们在外层循环将其切分为若干个微批次 (Micro-batches)。
- **通信原语的改变**： 在 PP 中，我们不再使用 `All-Reduce` 或 `All-Gather` 这种需要全局同步的集合通信原语，而是使用极其底层的**点对点通信 (Point-to-Point, P2P)**：
	- `dist.recv(tensor, src=prev_rank)`：阻塞式等待上一层传来的激活值。
	- `dist.send(tensor, dst=next_rank)`：计算完毕后，将输出激活值点对点直接推给下一层。

视频中演示的代码是一套极其脆弱的“同步”流水线：`send` 和 `recv` 会阻塞当前线程，依然存在严重的资源闲置。在真实的 Megatron-LM 或你日后要手写的系统级代码中，必须引入**计算与通信重叠 (Communication & Computation Overlap)**： 使用异步句柄（如 `dist.isend` 和 `dist.irecv`），在 GPU 进行当前微批次的矩阵乘法（计算密集型）的同时，让后台的 DMA 引擎去通过网卡发送上一个微批次的数据（IO 密集型）。在底层 C++ 层面，这涉及对多个 CUDA Stream（流）的并发控制。

## 编译器抽象、硬件演进与生态抉择

讲师跳出了 PyTorch 繁琐的底层通信逻辑，展示了 Google Jax/TPU 生态的解法，并讨论了为什么随着大模型的发展，底层硬件和软件编译器正在发生剧变。

手动管理 `Rank`、计算张量切片边界、手写 `send`/`recv` 的过程不仅极其枯燥，而且极易触发难以调试的死锁或 Segmentation Fault。为了让人类工程师更专注于模型算法，我们需要编译器来接管物理拓扑的调度。

写 PyTorch 分布式代码就像是手动控制内存指针（类似于 C 语言的 `malloc` 和并发 `mutex`）；而使用 Jax 的 `shard_map` 或 TPU 集群，就像是写高阶的声明式语言（比如 SQL），你只需要声明“把这个维度切在设备上”，底层的编译器（XLA）会自动帮你生成极其优化的集合通信指令，甚至自动处理流水线气泡。

深度学习的负载与传统 CPU 程序有着本质区别：由于神经网络的计算图 (Computation Graph) 在运行前是 100% 静态确定的（没有突发的 `if-else` 分支干扰），编译器理论上可以在编译期（AOT, Ahead-Of-Time）找到最优的数据流布局。这也是为什么 Groq、Cerebras 等新兴 AI 芯片抛弃了传统 GPU 复杂的缓存一致性协议，选择直接把巨大的 SRAM（静态随机存取存储器）做在芯片上，用纯粹的数据流 (Dataflow) 来驱动计算，试图从物理根源上彻底打破“内存墙”
